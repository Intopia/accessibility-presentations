What is a character counter widget?

Character counters helps users know how much text they can enter when there is a limit on the number of characters.

Ideally, these components provide users with live feedback to inform them about the following:

  • How many characters are left.
  • When they have reached the character limit.

The USWDS character counter is a great example.

Starting with people and their needs

When creating any new component, it is important to understand how it could impact different types of users.

I like to group people based on similar needs:

  1. People with varied learning or cognition.
  2. People with reduced colour vision.
  3. People with reduced vision.
  4. People with reduced movement.
  5. People with no vision.

The order?

I prefer to start with people with varied learning or cognition.

If the component does not work for these people, it has fundamental content or UX issues that need to be fixed before conducting any other testing.

I then focus on the people with reduced colour vision and reduced vision for all visual aspects of the component.

And then focus on people with reduced movement and no vision for the more technical aspects of the component.

Some common questions?

For each group, there are some common questions that can be asked for any component to help determine possible tests.

1. People with varied learning or cognition

  1. Is the purpose of the component clear?
  2. Is the component easy to use?
  3. If additional instructions are required, are they clear?
  4. If users make a mistake, can they quickly and easily recover?
  5. Is there any complex language used associated with the component?
  6. Are there any animations or movements that could distract the user?

2. People with reduced colour vision

Does the component:

  1. Have sufficient colour contrast?
  2. Use alternative methods to display colour information?

3. People with reduced vision

Sometimes, these people may need to customise the user interface or magnify the display.

Is the component operable/understandable:

  1. When the text is scaled to 400%?
  2. When the overall layout is scaled to 400%?
  3. When aspects of the component are magnified?
  4. When displayed in low contrast?
  5. When displayed in different viewport sizes?

4. People with reduced movement

Sometimes, these people may rely on keyboard interactions or voice recognition software.

  1. Does the component need to receive focus?
  2. Is focus managed into, within and out of the component?
  3. Does focus order follow a meaningful sequence?
  4. Can all actions be executed using keystrokes only?
  5. Can all actions be executed using voice controls?
  6. Are keystrokes intuitive for keyboard-only users?
  7. If focus is relevant, are all visible focus states clearly defined?

5. People with no vision

These people may rely on screen reader software.

  1. Does the component have an accessible name?
  2. Does the accessible name clearly define the component's purpose?
  3. Does the component have a role?
  4. Are all of the component's possible states defined?
  5. If a value can be added within the component, is it exposed?
  6. If the component has any dynamically added content, is this announced at the appropriate time?

What makes an accessible character counter?

What about when people are interacting with a character counter?

All users:

  • The overall component is easy to use and understand
  • All labels, instructions and error messages are persistent
  • Users are provided with feedback on the number of characters remaining
  • Users are provided with feedback when character number reached

All sighted users:

  • The form control has a visible text label

People with varied learning or cognition

  • All labels, instructions and error messages are clearly written and presented

People with reduced vision

  • All important information is aligned to the left to avoid being missed when enlarged

People with reduced vision and reduced colour vision

  • All aspects of the component have acceptable colour contrast (avoid placeholders)

People with reduced movement

  • All functionality is keyboard accessible
  • Focus states are clearly visible
  • Focus order is intuitive

People with no vision

  • The form control has an accessible name
  • All labels, instructions and error messages are programmatically associated with the form control
  • Users are provided with feedback on the number of characters remaining
  • Users are provided with feedback when character number reached

Marking up the basic character counter

What are the visual aspects of this component?

Step 1:

Add a <label>:

<label>Add a comment</label>
Step 2:

Add the instructions:

<label>Add a comment</label>
<span>Enter up to 100 characters</span>
Step 3:

Add the <textarea>:

<label>Add a comment</label>
<span>Enter up to 100 characters</span>
<textarea></textarea>
Step 4:

Programatically associated the label with the <textarea> using matching for and id values:

<label for="b1">Add a comment</label>
<span>Enter up to 100 characters</span>
<textarea id="b1"></textarea>
Step 5:

Programatically associated the instructions with the <textarea> using matching aria-describedby and id values:

<label for="b1">Add a comment</label>
<span id="a1">Enter up to 100 characters</span>
<textarea aria-describedby="a1" id="b1"></textarea>
Step 6:

Add an on-screen message character count message:

<label for="b1">Add a comment</label>
<span id="a1">Enter up to 100 characters</span>
<textarea aria-describedby="a1" id="b1"></textarea>
<span>27 characters left</span>

The character value would be dynamically updated using JavaScript as each character is added.

The problem is that this would be too verbose for screen reader users - possibly announcing information as they are typing.

Ideally, we need to hide this information from screen readers and provide them with less verbose updates.

Step 7:

We can make sure this information is not presented to screen readers using aria-hidden="true".

<label for="b1">Add a comment</label>
<span id="a1">Enter up to 100 characters</span>
<textarea aria-describedby="a1" id="b1"></textarea>
<span aria-hidden="true">27 characters left</span>
Step 8:

We could then add a screen reader-only message that will be less verbose.

<label for="b1">Add a comment</label>
<span id="a1">Enter up to 100 characters</span>
<textarea aria-describedby="a1" id="b1"></textarea>
<span aria-hidden="true">27 characters left</span>
<span>30 characters left</span>
Step 8:

This information can be hidden off-screen so that it is only available to screen readers.

<label for="b1">Add a comment</label>
<span id="a1">Enter up to 100 characters</span>
<textarea aria-describedby="a1" id="b1"></textarea>
<span aria-hidden="true">27 characters left</span>
<span class="hidden">30 characters left</span>

But how do we make this information ‘less verbose’?

There are two ways to tackle the problem:

Option 1:

We can set this message with aria-live="polite" so that the character count is announced at the next gracefull opportunity.

<label for="b1">Add a comment</label>
<span id="a1">Enter up to 100 characters</span>
<textarea aria-describedby="a1" id="b1"></textarea>
<span aria-hidden="true">27 characters left</span>
<span aria-live="polite"
  class="hidden">30 characters left</span>
Option 2:

We could use JavaScript Debounce to announce the character count after a pause in user typing.

What is debounce?

Debouncing is a programming technique that helps to improve the performance of web applications by limiting the frequency of function calls.
