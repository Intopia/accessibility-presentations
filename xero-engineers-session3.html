What is a component?

Components are self-contained UI concepts.

Simple interactive component examples:
  • Links
  • Buttons
  • Form controls and their associated labels
Complex interactive component examples:
  • Sortable tables
  • Accordions
  • Dropdown menus
  • Date pickers

Start with people!

You can start by looking at different groups of people and their needs.

Using people means you can focus on one user type and explore how they may need to interact with the component.

Five types of users:
  • People with a cognitive impairment or neurodivergence.
  • People with a colour vision deficiency.
  • People with low vision (e.g. magnifier users)
  • People with limited mobility (e.g. keyboard users)
  • People with limited or no sight (e.g. screen reader users)

Why are the groups of people in this order?

We should always start with the most basic concepts: can people understand the concept?

If not, there is no point in going deeper, as your results will be polluted.

We can then review the design layer - how the concept is perceived.

And finally, we can review the technical aspects - how assistive technologies can use the concept.

Why test keyboard users before screen reader users?

Screen reader users require the same basic functionality as keyboard users - with some additional criteria.

1. People with a cognitive impairment or neurodivergence

  • Is the purpose of the component clear?
  • Is the component easy to use?
  • If additional instructions are required, are they clear?
  • If users make a mistake, can they quickly and easily recover?
  • Is there any complex language associated with the component?
  • Are there any animations or movements that could distract the user?

These questions are UX problems, not Engineering problems.

However, designers do not always consider all of these questions.

Then, it is left up to engineers to build complex components or layouts that should never exist.

2. People with a colour vision deficiency

Does the component:
  • Have sufficient colour contrast?
  • Use alternative methods to display colour information?

These questions are UI problems, and should be resolved within the XUI design system.

3. People with low vision

Let’s meet Cammie, a ZoomText user - who sometimes inverts the screen colours to make information easier to see.

Is the component operable
  • When the text is scaled to 400%.
  • When the overall layout is scaled to 400%.
  • When aspects of the component are magnified.
  • When displayed in low contrast.
  • When displayed in different viewport sizes.

These problems need to be solved by designers and engineers.

The design should consider enlargement and magnification. Engineers must consider these problems when building.

Magnification activity:
  • How does the concept work when the overall layout is scaled to 400%?
  • How does the concept work when the text is scaled to 400%?

4. People with limited mobility

Sometimes, these people rely on keyboard interactions or voice recognition software.

Let’s meet Judith, a head-wand user - who also interacts with technology via the equivalent of a keyboard interface.

Keyboard use activity:
  • Can all actions be executed using keystrokes only?
  • Are keystrokes intuitive for keyboard-only users?
Focus indicator activity:
  • If focus is relevant, are all visible focus states clearly defined?
Focus order activity:
  • Does focus order follow a meaningful sequence?
Focus management activity:
  • Does the component need to receive focus?
  • Is focus managed into, within and out of the component?

5. People with limited or no sight

  • Does the component, or interactive elements within, have meaningful accessible names?
  • Does the component have relevant roles, states and values defined?
  • Are dynamic content changes announced at the appropriate time?
Screen reader activity:
  • Checking the accessible names.
  • Checking roles.
  • Checking states.
  • Checking dynamic changes.
