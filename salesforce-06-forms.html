Accessible forms for designers

Forms are everywhere — signing up, logging in, contacting support, making a payment.

If a form isn’t accessible, it can stop someone from completing a task entirely.

Designing better forms helps everyone, especially people using screen readers, keyboard navigation, or with cognitive differences.

Use Clear, Visible Labels

All form controls need to have a visible label.

That includes: text inputs, radio buttons, checkboxes, dropdowns, and text areas.

Labels help people identify the purpose of each field.

When labels are clear, people make fewer errors and complete forms more confidently.

Good labels should be short, descriptive, and written in plain, familiar language.

Use them consistently across your product.

Position labels close to the form field — either above or to the left, depending on your layout.

For radio buttons and checkboxes, the label usually sits to the right.

For example, on this registration flow, all labels are clearly written, associated, and easy to see.

Example: Use ‘Register for free’ flow at https://www.salesforce.com/plus

Avoid using placeholder text as a substitute for labels.

Placeholder text often has poor contrast and disappears when users type — which means they can’t refer back to the instructions later.

Always use persistent, well-placed, and clearly written labels.

They make forms easier for everyone.

Use Fieldset and Legend for Grouped Options

There may be times when we need a short, descriptive label above:

  • A group of form questions.
  • A group of radio buttons.
  • Or a group of checkboxes.

While this descriptive information is often referred to as another label, they are technically called a 'legend'.

A legend provides a caption for a group of form controls.

When correctly marked up, legends provide additional and important context for the group to screen reader users.

As focus moves to the first form control in the group, the legend is announced along with the label.

In the following example, screen readers would announce the first field as: "Delivery address. Address, edit."

As a designer, it is important to describe a legend correctly in our annotations.

This will allow our engineers to apply the appropriate markup.

Provide Helpful Instructions

Instructions can take two formats.

The first is form level instructions that are applied to the entire form.

And the second one is field level instructions which are applied to individual form fields.

Form level instructions help people to understand whether this form is right for them.

They help them understand what fields they'll need to fill in, and any instructions or requirements that apply to the entire form.

An example would be where all fields were identified as mandatory, unless otherwise marked.

Field level instructions indicate any restrictions or requirements that apply to an individual form field.

They help users understand if a field is required or optional, or that a specific format is needed.

For example, in this registration form, the field level instructions are used to inform people how many questions the form has, and what steps to follow.

Example: https://www.salesforce.com/form/signup/freetrial-salesforce-starter/?d=pb

The field level instructions indicate what the form is about and how many questions it contains.

For any instructions, we want to use plain language and an easy to see format.

For example, an asterisk is commonly used to mark fields as required.

However, it can easily be missed by people with low or no vision, or misunderstood by people with varied cognition.

Instead, we recommend being clear and explicit, such as marking required fields with a red 'required' in parentheses.

For specific formats, any requirements or restrictions should be made clear to eliminate doubt or confusion.

In this example, the date of birth field has the instruction text “DD/MM/YY”, indicating the date format.

And the Account name field has the instruction text which says, “This can be your username or email”.

As with labels, it's important that any instructions are persistent and not included as placeholder text.

That way, they can always be referred to as needed.

The same goes for not including instructions as error messages.

For example, in this same Salesforce form, there is no indication of what is a required field until an error occurs.

Ideally, we want to let users know this information before they trigger an error.

Example: https://www.salesforce.com/form/signup/freetrial-salesforce-starter/?d=pb

Required Fields

Designers need to make sure required fields are clearly identified — not just with color.

There are four methods that can be used to visually identify required fields:

Method 1: Asterisk symbol on required fields

The asterisk symbol can be used to visually define required fields.

If this method is used, a key needs to be presented at the top of the form to help explain the symbols.

However, the asterisk is a small indicator that can be missed by users with poor eyesight.

Method 2: (Required) wording on required fields

The word ‘(Required)’ can be used to visually define required fields.

<label for="name">
  Full name
  <span>(Required)</span>
</label>
<input type="text" id="name" required>

This additional content can be styled in a different colour to make it stand out.

Method 3: ‘All fields required’ statement

If all fields in the form are required, users can be presented with a statement such as: ‘All fields in the form below are required’

This message must be placed before the form so that assistive technology users can read this information before entering the form.

Using this method, individual form fields do not have to be visually identified as ‘Required’.

Method 4: ‘All fields required unless marked optional’ statement

If most of the form fields are required, you could flag optional fields only.

A message would need to be presented before the form along the lines of: ‘All fields in the form below are required unless marked OPTIONAL’.

Then, all optional fields would need a visual flag.

Make Errors Easy to Find and Fix

If someone makes a mistake, help them fix it quickly.

Show errors clearly, using both accessible color and text.

Place the message near the field that needs attention.

Tell the user what went wrong and how to fix it, like “Email address is missing the @ symbol.”

This Salesforce sign-up form uses clear error messages that tell the user directly how to fix the error, and are easy to see due to the use of colour and symbols.

Example: https://www.salesforce.com/au/form/signup/sales-ee/?d=topnav2-btn-ft

Remember: do not programmatically disable the submit control.

Users may need to rely on being able to submit in order to trigger validation errors.

Be Careful with Placeholders

The placeholder text isn’t a label.

It’s low contrast, disappears as you type, and can confuse users.

This can present issues for people with memory or cognitive issues.

Some patterns move the placeholder text out of the form field when a user begins to type, such as on this form.

This is a better practice than regular placeholders, but is less ideal than persistent label.

The text is still low contrast, and the movement may confuse users.

Example: https://www.salesforce.com/au/form/signup/sales-ee/?d=topnav2-btn-ft

Use placeholder text only for short examples — and never for essential instructions.

Conclusion

Accessible forms are just good design.

Clear labels, helpful feedback, and consistent structure make forms easier for everyone to use — with or without a disability.
