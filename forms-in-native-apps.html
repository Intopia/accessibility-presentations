Part 1. Screen reader modes on the web

On desktop web, screen readers like NVDA and JAWS operate in two modes:

Browse mode

Lets users read through the whole page, line by line and element by element.

Forms mode (JAWS)/ Focus mode (NVDA)

Triggered when entering form controls, or manually toggled.

In forms mode, keystrokes are sent directly to the focused control.

Screen readers usually only announce the control itself and any information that is programmatically associated with it.

Anything not programmatically tied to a form control can effectively become invisible while the user is in forms mode.

This is why programmatic association is non-negotiable on the web.

Without these connections, users may miss essential information!

Some common methods include:

Labels: via matching for and id values:

<label for="aaa">Email</label>
<input id="aaa" type="text">

Instructions and errors: via aria-describedby:

<label for="bbb">Email</label>
<span id="ccc">Use your full email address</span>
<input id="bbb" aria-describedby="ccc">

Invalid fields: via aria-invalid="true":

<label for="fff">Email</label>
<input id="fff" aria-invalid="true">

Required fields: via aria-required="true" or required:

<label for="hhh">Email</label>
<input id="hhh" required>

Radio group headings: via <fieldset> and <legend>

<fieldset>
  <legend>Do you like boats?</legend>
  ...
</fieldset>

Part 2. Why mobile feels different

Mobile screen readers (VoiceOver, TalkBack) do not have forms mode.

Users move between one element at a time using swipe or touch.

They can swipe through all focusable items on the screen.

This includes labels, instructions and error messages if those items are exposed to accessibility.

This means users are likely to encounter nearby text, even if it’s not formally associated.

This creates the illusion that “mobile forms don’t really need programmatic association.”

But that illusion is misleading.

Let’s look at some examples to help explain why programmatic association is still essential on mobile.

Example 1: Touch exploration

A screen reader user taps directly on a form field that has a visible label and instructions.

Only programmatically attached information is announced.

In the worst case, if the label and instructions are not programmatically associated, they will get:

“Text field.”

If the label is programmatically associated but the instructions are not, they will get:

“Phone number. Text field.”

If the label and instructions are programmatically associated, they will get:

“Phone number. Include your area code. Text field.”

Users could swipe up or down to find out more, but they should not have to.

Touch exploration is incredibly common — and it means that we must prgrammatically associated key info.

Example 2: Swiping into a field with an error message below it.

Imagine an error message positioned after a field but it has not been programmatically associated.

A screen reader user may hear the following:

Swipe 1 - Text label: “Email.”

Swipe 2 - Form field: “Text field.”

They may even hear the following if the label has been associated:

Swipe 1 - Text lable: “Email.”

Swipe 2 - Form field: “Email. Text field.”

But they will not be aware that there is an error message directly below the field.

So, they will not know that the field is currently in a state of error.

And, they will not have any information that may help them resolve the error.

Ideally, they should hear the following when swiping to the field:

“Email. Text field. Must be a valid email address.”

Part 3. Web vs iOS: shifting your mental model

1. No structural hierarchy

In HTML, there is a strong hierarchy of information.

Elements combine to form a meaningful, navigable document outline.

For example:
  • Headings form a nested tree.
  • Landmarks define regions of the page.
  • Lists introduce grouping and order.
  • Tables provide grid relationships.

The DOM provides the structural hierarchy, and the accessibility tree is generated directly from that structure.

In iOS, there is no inherent hierarchy of information.

VoiceOver receives a flat list of elements unless you explicitly group them or control their reading order.

2. No semantics

HTML elements have built-in semantics.

Each element carries a native role and purpose that browsers and assistive technologies understand.

<h1>This is a heading<h1>

In iOS, semantics are not built in.

They must be added manually through accessibility modifiers.

Examples of accessibility modifiers:
  • .accessibilityAddTraits(.isHeader)
  • .accessibilitySortPriority()
  • .accessibilityRole(.button)
  • .accessibilityHidden(true)
Text("This is a heading")
  .font(.title2.weight(.semibold)) // Size
  .accessibilityAddTraits(.isHeader) // Heading
  .padding(.bottom, 4)

3. No form structure

On the web, HTML provides three levels of built-in form structure:

Level 1: Form controls

The elements users interact with (input, textarea, select, checkbox, etc.)

Level 2: Form fields

A combination of:

  • The label
  • The control
  • Instructions (optional)
  • Required indicators (optional)
  • Error messages (optional)
Level 3: Form groups

Sets of related fields grouped under a shared heading, announced together by screen readers via <fieldset> and <legend>.

In iOS only the lowest-level concept exists — controls (TextField, Button, Picker, Toggle, etc.).

But:
  • There is no “form field” concept.
  • There is no container that binds label + control + instructions + error.
  • There is no <fieldset> or semantic form grouping.
Developers must manually assemble the entire experience:
  • A Text view (visible label)
  • A TextField or other control
  • A Text view for instructions or errors
  • Optional grouping using accessibility containers

4. No semantic <label> element

In the web world, we have the <label> element, which is used with form fields to provide a programmatically associated label.

In SwiftUI/iOS, there is no semantic <label> element.

There is only:
  • Visual text (Text).
  • Interactive controls (TextField, SecureField, etc).
Unlike the web:
  • You cannot bind a visual Text element to a form control.
  • There is no equivalent of <label for="id">.
  • iOS does not infer any relationship between a Text and a TextField.

This means all accessibility information must be applied directly to the control itself, including:

  • Accessible name
  • Required state
  • Instructions
  • Error messages

5. No semantic states

Another major difference between the web and iOS is how form field states are handled.

On the web: states are semantic.

You can mark a field as being in a particular state using explicit accessibility properties:

  • Error state: aria-invalid="true"
  • Required state: required or aria-required="true"

These states are semantic, machine-readable, and appear as distinct items in the accessibility tree.

In iOS/SwiftUI: states are not semantic.

iOS has no equivalent properties for:

  • “invalid”
  • “required”

There is no built-in way to declare a field as required or invalid using accessibility APIs.

Instead, everything must be expressed through text.

  • Error messages: “Error: You must use a valid email address.”
  • Required fields: “Email, required.”

6. A shift in thinking

If you come from a web background, native languages like SwiftUI and UIKit can feel less structured and less guided.

HTML gives you a well-defined hierarchy and built-in semantics.

In contrast, iOS requires you to build structure and meaning intentionally.

It’s not wrong — it just involves shifting your mental model.

Part 4. SwiftUI language basics

The following examples use SwiftUI because the code reads more like English and is less intimidating if you’re not a developer.

UIKit is still widely used, but the accessibility behaviour is the same at a conceptual level.

Before we look at some form examples, here is a breakdown of SwiftUI structure.

Example 1:

Text("Testing forms")
  .font(.largeTitle.weight(.bold))
  .accessibilityAddTraits(.isHeader)
  .padding(.bottom, 8)
// View type: Equivalent to an HTML element

Text()
// Initialiser argument: What the view displays

Text("Testing forms")
// Modifiers:

Text("Testing forms")
  .font(.largeTitle.weight(.bold))
  .accessibilityAddTraits(.isHeader)
  .padding(.bottom, 8)
// Modifier parameters: like CSS property values

Text("Testing forms")
  .font(.largeTitle.weight(.bold))
  .accessibilityAddTraits(.isHeader)
  .padding(.bottom, 8)

Example 2:

TextField("Phone", text: $phone)
// View type: Equivalent to an HTML input

TextField()
// Initialiser arguments: placeholder + binding

TextField("Phone", text: $phone)
// Placeholder text: Shown when the field is empty

TextField("Phone", text: $phone)
// Binding: Updates the variable as the user types

TextField("Phone", text: $phone)

Part 5. Adding instructions and errors in iOS

Adding an accessible name

Let’s start with a visible text label above a form field.


// Visible text label
Text("Phone")
  .font(.headline)

// Form field
TextField("Phone", text: $phone)

We can add an accessible name to a text field in two ways.

1. Use the placeholder text

// Visible text label
Text("Phone")
  .font(.headline)

// Form field
TextField("Phone", text: $phone)

This is acceptable practice in SwiftUI as long as no additional information is needed.

In HTML the placeholder has significant accessibility issues:

  • It cannot act as an accessible name reliably.
  • It creates usability and accessibility issues.
  • It is not persistent.
  • It conflicts with semantic labels.

In SwiftUI/iOS the placeholder can serve as the accessible name — but only when a visible text label is also present.

Just like in HTML, the placeholder must never be the only label.

  • It becomes the accessible name by default.
  • No semantic conflict (SwiftUI has no <label> element).
  • It is not misleading because the visible label is persistent.
  • It avoids most of HTML’s structural problems.
  • It aligns with iOS design conventions.
2. Use a custom accessibility label

Useful when you need to add extra information — such as “required”.

If you add an explicit accessible name to .accessibilityLabel, it will override the placeholder and be used as the accessible name.

// Visible text label
Text("Phone")
  .font(.headline)

// Form field
TextField("Phone", text: $phone)
  .accessibilityLabel("Phone, required")

Be aware that the placeholder is still used internally by SwiftUI as the TextField’s editing label, even when you override the accessible name.

SwiftUI exposes two separate concepts to VoiceOver:

  • The accessible name (from accessibilityLabel)
  • The editing label (derived from the placeholder)

When the strings differ, VoiceOver may announce both.

To avoid duplication, use an empty placeholder string ("") when providing a custom accessibilityLabel.

// Visible text label
Text("Phone")
  .font(.headline)

// Form field
TextField("", text: $phone)
  .accessibilityLabel("Phone, required")

There is no need for the following, as it offers no extra value — you may as well use the placeholder.

// Visible text label
Text("Phone")
  .font(.headline)

// Form field
TextField("Phone", text: $phone)
  .accessibilityLabel("Phone")

But how do we add instructions and error messages?

We’ll look at two approaches to associating instructions and errors with form fields.

Method 1: accessibilityHint

accessibilityHint allows you to attach a short description to the control.

It was originally designed for interaction guidance, not form-related instructions or errors.

Examples of interaction guidance
  • “Swipe left to reveal more items”
  • “Double tap to enlarge”

But accessibilityHint is often used to attach simple instructions.

// Visible text label
Text("Phone")
  .font(.headline)

// Form field
TextField("Phone", text: $phone)
  .accessibilityHint("Include your area code")
Pros:
  • Easy to implement
  • Reads automatically when the field is focused if “Speak Hints” is on.
Cons:
  • Can be turned off (“Speak Hints”)
  • Intended for interaction guidance, not business logic
  • One long, unstructured string
  • Not suitable as the only place for required information or errors

Method 2: UIAccessibilityCustomContent

UIAccessibilityCustomContent allows you to attach structured metadata (key–value) to a control.

  • Instructions
  • Error messages
// Visible text label
Text("Phone")
  .font(.headline)

// Form field
TextField("Phone", text: $phone)
  .accessibilityCustomContent(
    "Instructions", "Include your area code"
  )

You can also provide multiple pieces of supporting information.

// Visible text label
Text("Phone")
  .font(.headline)

// Form field
TextField("Phone", text: $phone)
  .accessibilityCustomContent(
    "Instructions", "Include your area code"
  )
  .accessibilityCustomContent(
    "Error", "Must be a valid area code"
  )
Pros:
  • Structured key/value pairs
  • Supports multiple types of information
  • Ideal for grouping supporting text in a predictable way
  • More semantic and flexible than a single hint
Cons:
  • By default, this content is only available through the rotor
  • Not all users know how to use the rotor
  • Not guaranteed to be announced when the field receives focus
  • VoiceOver always reads the key aloud (e.g., “Instructions”)

So, both methods have potential issues

Method 1:

If users turn off hints, instructions and errors may be missed.

Method 2:

Supporting text may require the rotor unless marked as high importance.

A solution?

One solution is to use UIAccessibilityCustomContent with importance: .high.

When custom content is marked as high importance, VoiceOver often includes it in the default announcement.

  • No rotor required (in most cases)
  • Less dependent on hints
  • The content becomes more predictable and discoverable

It promotes supporting text into the main VoiceOver announcement, while preserving its key–value structure.

// Visible text label
Text("Phone")
  .font(.headline)

// Form field
TextField("Phone", text: $phone)
  .accessibilityCustomContent(
    "Instructions",
    "Include your area code",
    importance: .high
  )

The resulting announcement will include the key and value:

“Phone. Instructions. Include your area code. Text field.”

There are some possible exceptions where high-importance content may not be included:

  • Very long accessibility labels
  • Speak Hints off (some iOS versions treat .high similarly to hints)
  • Certain rotor navigation modes
  • Parent elements that override accessibility
  • Combined child elements

There is a third, more complex approach. But before we proceed:

Label recommendations

1. Use the placeholder as the accessible label

When no additional information is required, the placeholder is acceptable provided a visible text label is also present.

TextField("Phone", text: $phone)
1. Use accessibilityLabel when additional information is needed

If you need to convey extra meaning (such as required fields), provide a custom accessible name.

.accessibilityLabel("Phone, required")

Instruction and error recommendations

1. Use accessibilityHint for short, unstructured instructions

Hints present additional context without requiring a key, and read naturally when VoiceOver hints are enabled.

.accessibilityHint("Include your area code")
2. Use accessibilityCustomContent for structured information such as errors

Errors naturally fit a key–value structure and are more predictable when announced in this form.

.accessibilityCustomContent(
  "Error", 
  "Must be a valid phone number",
  importance: .high
)

  • Which approach feels more predictable?

Part 6: A third approach?

This method can be used on more complex custom layouts.

Instead of attaching information to a field, you can group text and fields into one accessibility element that includes:

  • The label
  • The text field
  • Any instructions
  • Any error message

When the user focuses this element, VoiceOver reads the combined accessibilityLabel.

This gives developers a way to guarantee that users always hear:

  • The label
  • Whether the field is required
  • Any instructions
  • Any error messages

In the following example the label, text field and error text might all be separate visuals on screen.

But VoiceOver treats them as one combined announcement.

Here is a simplified example - developers would normally add additional conditions.

container.isAccessibilityElement = true
container.accessibilityLabel = 
  "Email. Must be a valid email address."
container.accessibilityTraits = .textField

container.accessibilityActivationPoint = 
    textField.center

Let's look at this line by line.

container.isAccessibilityElement = true
isAccessibilityElement

"Treat this entire container as one single accessibility element."

container.isAccessibilityElement = true
true

Don’t treat the label, field, and error message as separate things. When VoiceOver lands on it, read everything at once.

container.accessibilityLabel = 
    "Email. Must be a valid email address."

This replaces the default label of the text field. You now have full control over what is spoken, in what order

This is like using aria-label on a non-native container to override what is spoken for everything inside it.

container.accessibilityTraits = .textField

Without this line, VoiceOver would treat the container as a static element (just text).

Users wouldn’t know they can enter data.

This is like adding a role to a non-native HTML element.

container.accessibilityActivationPoint = textField.center
This tells VoiceOver:

“When the user double taps this combined element, actually activate the real text field inside it.”

This approach is especially useful for complex components:

  • Floating labels
  • Stylised form components
  • Complex custom layouts
  • Inline instructions or examples

It gives developers full control over what VoiceOver says.
