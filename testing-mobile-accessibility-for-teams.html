Testing mobile accessibility for teams

Session 1-3

About this course

The aim

The course covers 21 accessibility topics.

Each topic includes one or more simple tests to help you identify potential accessibility issues and create accessible native apps.

You won’t need any complex tools such as Android Accessibility Scanner or Xcode.

Hands-on

I’ll introduce each topic, then we’ll run through all of the tests together.

How we’ll work
  • Sometimes I’ll explain the test and you’ll run it on your app, then report back.
  • Other times I’ll demonstrate first, then you’ll try it yourself.
  • We’ll move slowly and clearly so you leave confident to test on your own.

Types of tests

As we explore each topic, there will be one or more tests you can do. These fall into four categories:

Test type 1: Quick check

By eye only.

Test type 2: Settings check

Toggle an OS or app setting, then check by eye.

Test type 3: Measure

Take a screenshot and use a colour picker or contrast checker.

Test type 4: Tool check

Use the keyboard or an accessibility tool such as a screen reader or voice control.

Who these tests help

We’ll focus on the following broad categories of disability and user needs:

People with no vision (who may use a screen reader)
  • headings, buttons, images
  • navigation order
  • form labels, instructions, errors
  • focus management
People with reduced vision
  • increase contrast mode
  • smart invert mode
  • dark mode
  • dynamic text resizing
  • proximity
People with reduced colour vision
  • text colour contrast
  • non-text colour contrast
  • use of colour
People with reduced movement (may have orientation locked, or use voice control)
  • screen orientation
  • gestures
  • touch target size
  • keyboard
  • voice control
People with varied learning or cognition
  • reduced motion

Using WCAG

WCAG applies to native mobile apps via WCAG2ICT, which maps WCAG 2 to non-web software.

WCAG2ICT is a W3C Group Note that is informative. It is not normative and does not set requirements.

For this reason, we will also briefly review WCAG criteria that are relevant to each topic.

About native vs WebView content

We’ll focus on native app testing.

However, some teams also embed web pages via WebViews.

For each test, we’ll note how it behaves in a WebView.

Section 1
Visual and colour

1. Testing text colour contrast

Aim

To understand how to check whether text has sufficient colour contrast for readability.

Audiences affected
  • People with reduced colour vision
  • People with reduced vision
Test Measure
  1. Take screenshots in light and dark mode (if available).
  2. Use a colour picker to sample text and background.
  3. Measure the contrast ratio.
  4. Check: does the text meet the required ratio?
Note

If text sits on an image or gradient, sample a few points to confirm contrast still meets the required ratio.

Pass Measure
  1. Normal text has at least 4.5:1 contrast (below 18pt regular or 14pt bold).
  2. Large text has at least 3:1 contrast (18pt or larger, or 14pt bold and above).
WebView note

Needs web support: contrast comes from CSS rules; test with a browser contrast tool.

WCAG requirements
  • SC 1.4.3 Contrast (Minimum) (Level AA): The visual presentation of text and images of text has a sufficient contrast ratio of 4.5:1 normal, 3:1 large.
  • SC 1.4.6 Contrast (Enhanced) (Level AAA): The visual presentation of text and images of text has a contrast ratio of at least 7:1 normal, 4.5:1 large.

2. Testing non-text colour contrast

Aim

To understand how to check whether key interface components and their states have sufficient colour contrast to be distinguishable.

Audiences affected
  • People with reduced colour vision
  • People with reduced vision
What is a UI boundary?

These outer edges of UI compoments, such as button and form control borders or outlines.

What is a state indicator?

This is the state of any element that is selected or in focus.

Test Measure
  1. Take a screenshot.
  2. Sample the UI boundary or state indicator and its background using a colour picker.
  3. Measure the contrast ratio.
  4. Check: Does the boundary or state indicator meet the minimum 3:1 ratio?
Pass Measure
  1. The contrast ratio between the UI boundary or state indicator and its background is at least 3:1.
WebView note

Needs web support: borders/states via CSS; OS setting won’t fix weak CSS rules.

WCAG requirements
  • SC 1.4.11 Non-text Contrast (Level AA): The visual presentation of the following have a contrast ratio of at least 3:1.

12. Testing navigation order

Aim

To understand how to check whether the navigation (focus/reading) order follows a clear, logical sequence that matches the visual layout and task flow.

Audiences affected

  • People using keyboard or switch access (single-step navigation)
  • People with varied learning or cognition who benefit from predictable sequences
Test Tool check
  1. Turn on a screen reader.
  2. Ensure the focus highlight is visible.
  3. Use swipe right (forward) and swipe left (back) to move through items one at a time.
  4. Check: As you swipe through the page, is everything in the expected visual order?
Pass Tool check
  1. The swipe order is logical and intuitive.
WebView note

Needs web support: DOM order/focus management/tabindex must match visual flow.

  • SC 2.4.3: Focus Order (Level A): Keyboard users navigate content in a correct order.

Audiences affected

  • People who experience motion sickness
  • People with vestibular disorders

Test Quick check

Section 5
Forms and interaction

Forms on mobile

Understanding screen readers and their two modes (web context)

On desktop web, screen readers like NVDA and JAWS operate in two modes:

  1. Browse mode (reading mode)
    moves linearly through the whole page
  2. Forms mode (focus mode)
    triggered when entering form controls, or manually toggled

In forms mode, keystrokes are sent directly to the focused control.

Screen readers usually only announce the control itself and any information that is programmatically associated with it.

Anything not programmatically tied to a form control can effectively become invisible while the user is in forms mode.

This is why programmatic association is non-negotiable on the web.

Without these connections, users may miss essential information!

  • Labels: via matching for and id values
  • Instructions and errors: via aria-describedby
  • Radio group headings: via <fieldset> and <legend>
  • Invalid fields: via aria-invalid="true"
  • Required fields: via aria-required="true" or required

Why some teams think this is less critical on mobile

Mobile screen readers (VoiceOver, TalkBack) do not have forms mode.

Users move between one element at a time using swipe or touch.

They will only hear text if it is exposed as an accessibility element and they move focus to it.

This means users will naturally encounter nearby text, even if it’s not formally associated.

This creates the illusion that “mobile forms don’t really need programmatic association.”

But that illusion is misleading.

Let’s look at some examples to help explain why programmatic association is still essential on mobile.

Example 1: Touch exploration

A blind user drags a finger and lands directly on a form field.

Only programmatically attached information is announced. So, additional information or error messages would not be announced.

Without association:

“Phone number. Text field.”

With association:

“Phone number. Include your area code. Text field.”

Touch exploration is incredibly common — and it requires associated info.

Example 2: Swiping into a field with errors or instructions below it

Imagine an error message positioned after a field.

This error message would not be announced when the user swipes to the field.

This means they may not be aware of the error message and any information that may help them fill in the field successfully.

Without association:

“Email. Text field.”

With association:

“Email. Text field. Must be a valid email address.”

Input and focus

18. Testing touch target size

Aim

To understand how to check whether important actions have a large enough touch target and enough space around them to support accurate tapping.

Audiences affected
  • People with reduced movement
  • People with tremor or tics
  • People with large fingers
  • People with reduced vision who use screen magnification

Note on target size:

SC 2.5.8: Target Size (Minimum) (Level AA) requires either:

A target is at least 24 × 24 CSS px.

On native apps, treat CSS px as logical pixels:

  • iOS: 24 x 24 pt (point)
  • Android: 24 × 24 dp (density-independent pixel)

Or there is enough spacing that a 24-px diameter circle centred on the target does not intersect any other target.

However, you should aim for larger target sizes where practical: 44 × 44 pt (iOS) or 48 × 48 dp (Android).

“Hit slop” or “Expanded hit area”

The invisible tappable padding added around a control so it’s easier to tap.

Settings Show layout bounds (Android)

Settings → Developer options → Drawing → Show layout bounds

“Show layout bounds” displays the actual view boundary, including padding. This makes it easier to measure target size and spacing without using overlays.

Test Measure (iOS)
  1. Take a screenshot.
  2. Check:
    • Does the target meet or exceed the 24 × 24 px square? OR
    • Does the 24 px circle fit without touching neighbouring targets?
How to measure target size (iOS)
  1. In your editor, insert two overlays (use the provided PNGs at your device scale, e.g., 24@1x / 48@2x / 72@3x):
    • 24 × 24 px square (size check)
    • 24 px diameter circle (spacing check)
  2. For each button/target, centre the overlays on the control. If taps just outside the visible edges still activate it, include that space as part of the measurement.
Test Measure (Android)
  1. Turn on Layout Bounds.
  2. Take a screenshot.
  3. Check:
    • Does the target meet or exceed the 24 × 24 px square? OR
    • Does the 24 px circle fit without touching neighbouring targets?
Pass Measure
  1. The target is at least 24 × 24 px, OR
  2. A 24 px circle centred on the target does not overlap neighbouring targets.
WebView note

Content inside a WebView must meet the same requirements: targets of at least 24×24 CSS px or enough spacing. Add padding or spacing in CSS where needed.

WCAG requirements
  • SC 2.5.8: Target Size (Minimum) (Level AA): Make controls easier to activate.
  • SC 2.5.5: Target Size (Level AAA): Controls can be operated more easily, especially on touch screens.

19. Testing focus management

Aim

To understand how to check whether focus is managed in an intuitive way.

Audiences affected
  • People with reduced movement
  • People with reduced or no vision who use a screen reader or braille display
  • People with varied learning or cognition
Which actions may change focus?
  • Opening a modal, menu, or panel
  • Submitting a form
  • Revealing instructions or errors
  • Moving to a new screen
Test Settings check
  1. Turn on a screen reader.
  2. Trigger an action that changes the interface.
  3. Check:
    • Did focus move?
    • Was the focus movement necessary?
    • Was the movement logical and beneficial to the user?
Pass Settings check
  1. Focus was only moved when necessary.
  2. The movement was logical, beneficial and intuitive.

  2. SC 2.1.1: Keyboard (Level A): Everything can be done with a keyboard except freehand movements.
  3. SC 2.4.7: Focus Visible (Level AA): Users know which element has keyboard focus.

20. Testing keyboard

Aim

To understand how to check whether all functionality can be used via a keyboard.

Audiences affected
  • People with reduced movement
  • People who rely on a hardware keyboard or alternative input devices that emulate keyboard input
  • People using a keyboard in conjumction with a screen reader

Testing keyboard

When testing keyboard support on mobile devices, you can evaluate standard keystrokes using a bluetooth keyboard.

However, if you turn on a screen reader (VoiceOver on iOS or TalkBack on Android), the interaction model changes slightly.

Some keys may be repurposed for screen reader navigation, not for interacting directly with the UI.

1. Test keyboard access without the screen reader, to confirm actual keyboard operability.

2. Test with the screen reader, understanding that behaviour will be different and keystrokes may not reflect true keyboard accessibility.

Navigation (without a screen reader)

Navigating through the page
  • Use TAB / SHIFT + TAB to move focus through interactive elements.
  • Use ARROW keys when components support internal navigation.
Navigating menus & dropdowns
  • Open with ENTER.
  • Move with UP ARROW and DOWN ARROW.
  • Select with ENTER.
  • Close with ESC.
Navigating radio groups
  • Move within the group with ARROW keys.
  • Select with SPACE.
Interacting with links and buttons
  • Activate links with ENTER.
  • Activate buttons with SPACE or ENTER.
Test Tool check
  1. Connect an external keyboard.
  2. Check:
    • Are all controls operable via keyboard alone?
    • Is there a visible focus state?
    • Does focus follow a logical order?
Pass Tool check
  1. All controls are operable via keyboard alone.

21. Testing voice control

Aim

To understand how to check whether all functionality can be used via voice control.

Audiences affected
  • People with reduced motion
  • People who rely on voice contol to navigate and interact with controls
Common interactions using visible labels
  • Say: “Tap Email address”
  • Say: “Tap Subscription topic”
Common interactions using numbers
  • Say: “Show numbers”
  • Say: “Tap [number]
Settings Voice Control (iSO)

Settings → Accessibility → Voice Control

Settings Voice Access (Android)

Settings → Accessibility → Voice Access

Test Tool check
  1. Turn on Voice Control (iSO) or Voice Access (Android).
  2. Check:
    • All controls are operable by voice alone.
    • Do any duplicate or ambiguous labels exist?
Pass Tool check
  1. All controls are operable by voice alone.
  2. No duplicate or ambiguous labels exist.
WebView note

Needs web support: Label-in-Name: visible text must be in accessible name so “Tap X” works.

WCAG requirements
  • SC 2.5.3: Label in Name (Level A): The visual label for controls is a trigger for speech activation.
  • SC 4.1.2: Name, Role, Value (Level A): People using assistive technology understand all components.
  • SC 1.3.1: Info and Relationships (Level A): Information about content structure is available to more people.
