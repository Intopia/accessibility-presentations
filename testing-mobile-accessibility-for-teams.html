Testing mobile accessibility for teams

Session 1

Introduction

The aim of this course

The aim of this course is to give you a range of simple accessibility tests that you can perform on your apps.

These simple tests will help you determine whether your products are accessible.

The course covers 22 accessibility topics, each showing you how to run a range of simple tests — from quick visual reviews to settings and tool-based tests.

You won’t need any complex tools such as Android Accessibility Scanner or Xcode.

All tests can be done manually or by using built-in system settings on iOS or Android devices.

Types of tests

As we explore each topic, there will be one or more tests you can do. These fall into four categories:

Test type 1: Quick check

By eye only.

Test type 2: Settings check

Toggle an OS or app setting, then check by eye.

Test type 3: Measure

Take a screenshot and use a colour picker or contrast checker.

Test type 4: Tool check

Use the keyboard or an accessibility tool such as a screen reader or voice control.

These checks will help you identify and fix common accessibility issues early in the process.

Who this helps

We’ll focus on the following broad categories of disability and user needs:

  • People with no vision
  • People with reduced vision
  • People with reduced colour vision
  • People with reduced movement
  • People with varied learning or cognition
  • People with reduced or no hearing

Using WCAG

We will also briefly review WCAG criteria that are relevant to each topic.

WCAG applies to native mobile apps via WCAG2ICT, which maps WCAG 2 to non-web software.

Section 1
Visual and colour

1. Testing text colour contrast

Aim

To understand how to check whether text has sufficient colour contrast for readability.

Audiences affected
  • People with reduced colour vision
  • People with reduced vision
Test: Quick check
  1. Check: readability of the text. Is the text comfortably readable without strain?
Test: Measure
  1. Take screenshots in light and dark mode (if available).
  2. Use a colour picker to sample text and background.
  3. Measure the contrast ratio.
  4. Check: does the text meet the required ratio?
WCAG requirements
  • SC 1.4.3 Contrast (Minimum) (Level AA): The visual presentation of text and images of text has a sufficient contrast ratio of 4.5:1 normal, 3:1 large.
  • SC 1.4.6 Contrast (Enhanced) (Level AAA): The visual presentation of text and images of text has a contrast ratio of at least 7:1 normal, 4.5:1 large.

2. Testing non-text colour contrast

Aim

To understand how to check whether key interface components and their states have sufficient colour contrast to be distinguishable.

Audiences affected
  • People with reduced colour vision
  • People with reduced vision
Test: Quick check
  1. Check: visibility of borders and state indicators (e.g. active, focused, hovered, selected). Are they easy to see in each state?
Test: Measure
  1. Take screenshots in light and dark mode (if available).
  2. Use a colour picker to sample the border/indicator and the adjacent or underlying background colour, including pressed/selected/focused states.
  3. Measure the contrast ratio.
  4. Check: do components and states meet the required ratio? Ignore disabled components.
WCAG requirements
  • SC 1.4.11 Non-text Contrast (Level AA): The visual presentation of the following have a contrast ratio of at least 3:1.

3. Testing use of colour

Aim

To understand how to check whether colour is used as the only way of conveying important information, prompts, or feedback.

Audiences affected
  • People with reduced colour vision
Test: Quick check
  1. Check: Is there at least one additional cue besides colour to indicate the information or state (e.g. icon, text message, pattern, change in shape)?
WCAG requirements
  • SC 1.4.1: Use of Color (Level A): Color is not the only way of distinguishing information.

4. Testing increase contrast mode

Aim

To understand how to check whether text and interface elements remain clear and distinguishable when system contrast settings are increased.

Audiences affected
  • People with reduced vision
Test: Settings check
  1. Turn on Increase Contrast (iOS) or High contrast text (Android, if available).
    • iOS: Settings → Accessibility → Display & Text Size → Increase Contrast
    • Android: Settings → Accessibility → Display size and text → High contrast text
  2. Check: In both light and dark modes, are text, icons, and component boundaries easier to see with this setting enabled?
WCAG requirements
  • No direct criterion.

Pointers for developers

iOS
  • Prefer semantic system colours (e.g., UIColor.label, systemBackground; SwiftUI Color.primary/secondary) so iOS can auto-boost contrast.
  • Detect setting with UIAccessibility.isDarkerSystemColorsEnabled and observe UIAccessibility.darkerSystemColorsStatusDidChangeNotification.
  • For custom tokens, provide “darker” variants (stronger borders/separators; avoid ultra-low-alpha fills).
Android
  • Check AccessibilityManager.isHighTextContrastEnabled() (or Settings.Secure.ACCESSIBILITY_HIGH_TEXT_CONTRAST_ENABLED).
  • Use Material colour tokens (MaterialTheme.colorScheme.*); avoid hard-coded greys—let components render text/indicators.
  • High-contrast text mostly affects text; add higher-contrast borders/state layers for focus/selected states.

5. Testing smart invert mode

Aim

To understand how to check whether text, images, and interface elements remain clear and usable when system colour inversion settings are enabled.

Audiences affected
  • People with reduced colour vision
  • People with reduced vision
  • People sensitive to brightness or glare
Test: Settings check
  1. Turn on Colour inversion:
    • iOS: Settings → Accessibility → Display & Text Size → Smart Invert
    • Android: Settings → Accessibility → Colour inversion
  2. Check: Do text, icons, and images remain easy to see and correctly displayed (not “negative” or distorted)?
WCAG requirements
  • No direct criterion.

Pointers for developers

iOS
  • Mark views that must not invert (photos/videos) with accessibilityIgnoresInvertColors = true
  • Use sparingly so icons and illustrations that should adapt remain invertible.
  • Listen for UIAccessibility.invertColorsStatusDidChange and verify critical screens still look correct when Smart Invert toggles.
Android
  • No Smart Invert equivalent; Color inversion affects the entire display.
  • Test with inversion on and confirm key UI remains legible.
  • Provide a robust dark theme with accessible contrast so users don’t need inversion to see content.
  • Avoid fragile assets (e.g., dark-on-dark logos, anti-aliased halos); use vector icons and token-based colours that survive inversion.

6. Testing dark mode

Aim

To understand how to check whether text, icons, and components remain clear and usable when the system (or app) is set to dark mode.

Audiences affected
  • People with reduced vision (glare sensitivity, low-contrast issues)
  • People sensitive to brightness or who prefer darker themes for comfort
  • People using devices in low-light environments
Test: Settings check
  1. Enable dark mode:
    • iOS: Settings → Display & Brightness → Appearance: Dark (or Control Center)
    • Android: Settings → Display → Dark theme
  1. Check legibility: Is all text comfortably readable?
  2. Check non-text contrast: Are borders, dividers, states, and icons distinguishable from backgrounds?
  3. Check assets: Do logos/illustrations render correctly?
  4. Check colours: Do colours chosen for dark backgrounds still meet contrast needs for text/icons?
  5. Check states: Are selected/focused/disabled states still clearly distinguishable in dark mode?
WCAG requirements
  • No direct criterion.

Pointers for developers

iOS
  • Prefer semantic/dynamic colours (e.g., UIColor.systemBackground, label; SwiftUI Color.primary/secondary) instead of hard-coded hex values.
  • Provide dark-aware assets (Asset Catalog light/dark variants; SF Symbols/monochrome where possible); avoid dark-on-dark logos/halos.
  • Follow system theme changes (e.g., traitCollection updates) and verify elevation/overlays & states (selected/disabled/focus) remain clearly distinguishable.
Android
  • Use a Material 3 Dark theme (colorScheme), or DayNight and follow system; avoid hard-coded greys.
  • Supply -night resources for colours/illustrations as needed; prefer vector icons and token-based colours.
  • Verify contrast and states in dark (text, icons, borders, selected/disabled/focus) and ensure surfaces/elevation separate from the background.

Section 2
Text size and layout

7. Testing dynamic text resizing

Aim

To understand how to check whether text remains clear, readable, and functional when system font sizes are increased.

Audiences affected
  • People with reduced vision
Test: Settings check
  1. Increase text size:
    • iOS: Settings → Accessibility → Display & Text Size → Larger Text → turn on Larger Accessibility Sizes, then drag the slider near maximum (Dynamic Type)
    • Android: Settings → Accessibility → Display size and text → Font size → set to largest
  2. Check: Is any content truncated, overlapping, or cut off?
WCAG requirements
  • SC 1.4.4: Resize Text (Level AA): Text can be enlarged.
  • SC 1.4.10: Reflow (Level AA): Content can be enlarged without increasing line length.

Pointers for developers

iOS
  • Use Dynamic Type text styles (UIKit UIFont.preferredFont(forTextStyle:) with adjustsFontForContentSizeCategory = true; SwiftUI .font(.body) or .custom(..., relativeTo: .body)).
  • Avoid fixed-height containers; let text wrap/reflow (UIKit numberOfLines = 0, sensible Auto Layout; SwiftUI avoid fixed frames).
  • When space is tight, prefer stacking/reflow over shrinking—don’t rely on aggressive minimumScaleFactor as a crutch.
Android
  • Use sp for all text and rely on Material typography (Compose MaterialTheme.typography) so font scale is respected.
  • Keep layouts flexible (wrap_content, constraints); avoid fixed heights; allow wrapping; don’t ellipsize essential content.
  • Test at the largest font scale and fix any clipping/overlap by reflowing, stacking, or enabling scroll where appropriate.

8. Testing screen orientation

Aim

To understand how to check whether content remains clear, readable, and functional when the device orientation changes between portrait and landscape.

Audiences affected
  • People with reduced movement
Test: Settings check
  1. Ensure device rotation is enabled:
    • iOS: Control Center → Portrait Orientation Lock → Off
    • Android: Settings → Auto-rotate → On
  2. Rotate the device between portrait and landscape.
  3. Check: Does the layout reflow and remain fully usable in both orientations?
WCAG requirements
  • SC 1.3.4: Orientation (Level AA): Devices can be used in any orientation.

Pointers for developers

iOS
  • Support both orientations in Project/Info.plist (“Supported Interface Orientations”); design with Auto Layout + size classes; avoid forcing orientation unless essential.
  • On rotate, adapt layout (stack/reflow) and preserve state & focus; respond to traitCollection/size changes rather than hard-coding frames.
  • Verify safe areas/keyboard/modals don’t clip controls in landscape; keep primary actions reachable.
Android
  • Avoid locking android:screenOrientation; let UI adapt with ConstraintLayout/Compose and window size classes; only use -land resources when necessary.
  • Handle configuration changes: keep state via ViewModel/rememberSaveable and restore focus/scroll after rotation.
  • Test phones, tablets, and foldables; ensure system bars/FAB/IME don’t cover key actions in landscape.

Section 3
Elements and structure

9. Testing headings

Aim

To understand how to check whether headings are used and structured correctly so they are identifiable and navigable using screen readers.

Audiences affected
  • People with no vision who rely on a screen reader
  • People with reduced vision who use a screen reader or voice navigation
Test: Quick check
  1. Check: Do sections include clear, descriptive headings above the related content?
Test: Tool check – General
  1. Turn on a screen reader:
    • iOS: Settings → Accessibility → VoiceOver
    • Android: Settings → Accessibility → TalkBack
  2. Swipe through elements.
  3. Check: Are headings announced as “heading” when read by the screen reader?
Test: Tool check – iOS
  1. In Settings → Accessibility → VoiceOver → Rotor, ensure Headings is enabled.
  2. Activate the Rotor by placing two fingers on the screen and twisting as if turning a dial.
  3. Select Headings, then swipe up or down to navigate.
  4. Check: Are headings available and in the correct order?
Test: Tool check – Android
  1. Turn on a screen reader:
    • Android: Change Navigation granularity to Headings (swipe up or down until you hear “Headings”)
  2. Check: Can you move directly between headings using up/down gestures?
WCAG requirements
  • SC 1.3.1: Info and Relationships (Level A): Information about content structure is available to more people.
  • SC 2.4.6: Headings and Labels (Level AA): A page’s content is described in headings and labels

Pointers for developers

iOS
  • Mark real section titles as headings so VoiceOver can navigate
  • iOS (UIKit): accessibilityTraits.insert(.header).
  • SwiftUI: .accessibilityAddTraits(.isHeader)
Android
  • Mark real section titles as headings so TalkBack can navigate
  • Android (XML): android:accessibilityHeading="true" (API 28+)
  • AndroidX: ViewCompat.setAccessibilityHeading(view, true)
  • Jetpack Compose: Modifier.semantics { heading() }

10. Testing buttons

Aim

To understand how to check whether buttons are labelled and coded correctly so they are identifiable and usable with screen readers and voice control.

Audiences affected
  • People with no vision who rely on a screen reader
  • People with reduced vision who use a screen reader or voice navigation
  • People with reduced movement who rely on voice commands instead of touch
Test: Quick check
  1. Check: Does the button include a clear, descriptive visible label that communicates its purpose?
Test: Tool check – Screen reader
  1. Turn on a screen reader:
    • iOS: Settings → Accessibility → VoiceOver
    • Android: Settings → Accessibility → TalkBack
  2. Swipe through the interface.
  3. Check: Are buttons announced as “button” and do their names match their visible labels?
Test: Tool check – Voice control
  1. Turn on voice control:
    • iOS: Turn on Voice Control: Settings → Accessibility → Voice Control
    • Android: Turn on Voice Access: Settings → Accessibility → Voice Access
  2. Say “Tap Shop online”
  3. Check: is the control activated?
WCAG requirements
  • SC 4.1.2: Name, Role, Value (Level A): People using assistive technology understand all components.
  • SC 1.3.1: Info and Relationships (Level A): Information about content structure is available to more people.
  • SC 2.5.3: Label in Name (Level A): The visual label for controls is a trigger for speech activation.

Pointers for developers

iOS
  • Prefer native UIButton / SwiftUI Button; for custom views, add the button trait (UIKit accessibilityTraits = .button; SwiftUI .accessibilityAddTraits(.isButton)).
  • Keep the accessible name = visible label (meets Label in Name); don’t hide key words in a hint or swap in a different label.
  • Ensure buttons announce and perform a default action (activate on double-tap / Enter), and expose state (enabled/disabled, selected) appropriately.
Android
  • Prefer MaterialButton / Compose Button {}; for custom components, set button role/semantics (Compose Modifier.semantics { role = Role.Button }).
  • Keep the spoken label = visible text; avoid overriding with a different contentDescription when text is on screen; give icon-only buttons a clear action name.
  • Ensure buttons are focusable/clickable, activate with Enter/Space and Voice Access (“Tap X”), and reflect state (enabled/disabled, selected).

11. Testing images

Aim

To understand how to check whether decorative, informative, functional, and complex images are handled correctly, with appropriate alternatives.

Audiences affected
  • People with reduced or no vision who use a screen reader, text-to-speech, or a Braille display
Test: Settings check
  1. Turn on a screen reader:
    • iOS: Settings → Accessibility → VoiceOver
    • Android: Settings → Accessibility → TalkBack
  1. Check:
    • Decorative: Is it ignored?
    • Informative: Is a brief, meaningful description announced?
    • Functional: Is the destination or action announced?
    • Complex: Is a short alt announced plus access to a full on-screen description?
WCAG requirements
  • SC 1.1.1: Non-text Content (Level A): Non-text information is available to more people.

12. Testing navigation order

Aim

To understand how to check whether the navigation (focus/reading) order follows a clear, logical sequence that matches the visual layout and task flow.

Audiences affected
  • People with no or reduced vision who rely on linear navigation (screen readers, braille)
  • People using keyboard or switch access (single-step navigation)
  • People with varied learning or cognition who benefit from predictable sequences
Test: Tool check – Screen reader with focus highlight
  1. Turn on a screen reader:
    • iOS: Settings → Accessibility → VoiceOver
    • Android: Settings → Accessibility → TalkBack
  2. Ensure the focus highlight is visible.
  3. Use swipe right (forward) and swipe left (back) to move through items one at a time.
  1. Check: As you swipe through the page, is everything in the expected visual order?
  2. Check: As you swipe through modals, groups or forms, is the internal order logical?
  3. Check: As you swipe through sticky elements, do they appear in the expected order?
WCAG requirements
  • SC 2.4.3: Focus Order (Level A): Keyboard users navigate content in a correct order.
  • SC 1.3.2: Meaningful Sequence (Level A): The order of content can be understood by more people.
  • SC 1.3.1: Info and Relationships (Level A): Information about content structure is available to more people.

Section 4
Motion and sensory

13. Testing reduced motion

Aim

To understand how to check whether animations are coded so that motion can be stopped, reduced, or simplified based on user settings.

Audiences affected
  • Autistic people or people with ADHD who may find movement distracting
  • People who experience motion sickness
  • People with vestibular disorders
  • People who experience migraine
  • People at risk of seizures
Test: Quick check
  1. Check: If any animation auto-plays for longer than 5 seconds alongside other content, is there a clear Pause / Stop / Hide control?
Test: Settings check
  1. Turn on reduced motion:
    • iOS: Settings → Accessibility → Motion → Reduce Motion
    • Android: Settings → Accessibility → Display size and text → Remove animations (availability varies by device; some users rely on Developer options to disable animation scales)
  1. Check: When ‘Reduce Motion’/‘Remove animations’ is enabled, do auto-playing and interaction-triggered animations stop or reduce/simplify?
WCAG requirements
  • SC 2.2.2: Pause, Stop, Hide (Level A): Fewer users are distracted by content that updates or moves.
  • SC 2.3.1: Three Flashes or Below Threshold (Level A): Content does not trigger seizures.
  • SC 2.3.3: Animation from Interactions (Level AAA): Users are not harmed or distracted by motion.

14. Testing proximity

Aim

To understand how to check whether feedback and results (e.g., confirmations, error messages) appear close to the control that triggered them, so users don’t have to pan or search.

Audiences affected
  • People with reduced vision who use magnification
  • People with varied learning or cognition (reduced attention/working memory)
  • People with reduced movement (minimising extra navigation/panning)
Test: Quick check
  1. Trigger the action.
  2. Check: Does the modal/popover/dialog/notification appear adjacent to the trigger mechanism and within the current viewport?
Test: Settings check
  1. Turn on magnification:
    • iOS: Settings → Accessibility → Zoom → On
    • Android: Settings → Accessibility → Magnification → On
  2. Trigger the action.
  3. Check: with magnification enabled, does the modal/popover/dialog/notification appear near the trigger mechanism so it is easy to find?
WCAG requirements
  • No direct criterion.

15. Testing gestures

Aim

To understand how to check whether important actions can be completed with simple, single-pointer gestures or controls (not only multi-point or path-based gestures).

Audiences affected
  • People with reduced movement
  • People with tremor or tics
  • People who may rely on pointer devices, voice control, or switch access
  • People with reduced vision using magnification
Test: Quick check
  1. Open Google Maps.
  2. Check: Can you pinch to zoom in/out? (complex, multi-point gesture)
  3. Check: Can you double-tap to zoom in? (simple, single-pointer)
  4. Check: Is there a single-tap control to zoom out?
WCAG requirements
  • SC 2.5.1: Pointer Gestures (Level A): Let users operate touchscreens with one finger and reduced gestures.
  • SC 2.5.7: Dragging Movements (Level AA): Don”t rely on dragging for user actions.

Section 6
Input and focus

19. Testing touch target size

Aim

To understand how to check whether important actions have a large enough touch target and enough space around them to support accurate tapping.

Audiences affected
  • People with reduced movement
  • People with tremor or tics
  • People with large fingers
  • People with reduced vision who use screen magnification

Note on target size:

SC 2.5.8: Target Size (Minimum) (Level AA) requires either:

A target is at least 24 × 24 CSS px.

On native apps, treat CSS px as logical pixels:

  • iOS: 24 x 24 pt
  • Android: 24 × 24 dp

Or there is enough spacing that a 24-px diameter circle centred on the target does not intersect any other target.

However, you should aim for larger target sizes where practical: 44 × 44 pt (iOS) or 48 × 48 dp (Android).

“Hit slop” or “Expanded hit area”

The invisible tappable padding added around a control so it’s easier to tap.

Test: Quick check
  1. Identify the smallest or most crowded buttons on the screen.
  2. Tap each three times at normal speed without zooming.
  3. Check: Can you activate the intended button without hitting a nearby button by accident?
Test: Measure
  1. Take a screenshot.
  2. In your editor, insert two overlays (use the provided PNGs at your device scale, e.g., 24@1x / 48@2x / 72@3x):
    • 24 × 24 px square (size check)
    • 24 px diameter circle (spacing check)
  3. For each button/target, centre the overlays on the control. If taps just outside the visible edges still activate it, include that space as part of the measurement.
  1. Size check: Does the target meet or exceed the 24 × 24 square in both width and height?

If not:

  1. Spacing check: Does a 24 px circle centred on the target stay clear of neighbouring targets?
WCAG requirements
  • SC 2.5.8: Target Size (Minimum) (Level AA): Make controls easier to activate.
  • SC 2.5.5: Target Size (Level AAA): Controls can be operated more easily, especially on touch screens.

20. Testing focus management

Aim

To understand how to check whether focus is managed in an intuitive way.

Audiences affected
  • People with reduced movement
  • People with reduced or no vision who use a screen reader or braille display
  • People with varied learning or cognition
Test: Settings check
  1. Turn on a screen reader:
    • iOS: Settings → Accessibility → VoiceOver
    • Android: Settings → Accessibility → TalkBack
  2. Trigger the modal (tap the button).
  1. Check: When the modal is triggered, focus moves inside the modal (e.g., on the modal title or first actionable element).
  2. Check: When the modal is open, focus cannot move to background content.
  3. Check: When the modal is closed, focus returns to the trigger button.
WCAG requirements
  1. SC 2.4.3: Focus Order (Level A): Keyboard users navigate content in a correct order.
  2. SC 2.1.1: Keyboard (Level A): Everything can be done with a keyboard except freehand movements.
  3. SC 2.4.7: Focus Visible (Level AA): Users know which element has keyboard focus.

21. Testing keyboard

Aim

To understand how to check whether all functionality can be used via a keyboard.

Audiences affected
  • People with reduced movement
  • People who rely on a hardware keyboard or alternative input devices that emulate keyboard input
Navigating through the page
  • Use TAB / SHIFT + TAB to move focus through interactive elements.
  • Use ARROW keys when components support internal navigation.
Navigating menus & dropdowns
  • Open with ENTER.
  • Move with UP ARROW and DOWN ARROW.
  • Select with ENTER.
  • Close with ESC.
Navigating radio groups
  • Move within the group with ARROW keys.
  • Select with SPACE.
Interacting with links and buttons
  • Activate links with ENTER.
  • Activate buttons with SPACE or ENTER.
Test: Tool check
  1. Connect an external hardware keyboard (Bluetooth or USB).
  2. iOS: Enable Full Keyboard Access: Settings → Accessibility → Keyboard → Full Keyboard Access → On.
  1. Check: All controls are operable via keyboard alone (no touch required).
  2. Check: Focus is visible and follows a logical order (no unexpected jumps).
  3. Check: There is no keyboard trap — you can move focus away or close pop-ups using keyboard commands.
WCAG requirements
  • SC 2.1.1: Keyboard (Level A): Everything can be done with a keyboard except freehand movements.
  • SC 2.1.2: No Keyboard Trap (Level A): Keyboard users don't get stuck.
  • SC 2.4.3: Focus Order (Level A): Keyboard users navigate content in a correct order.
  • SC 2.4.7: Focus Visible (Level AA): Users know which element has keyboard focus.

22. Testing voice control

Aim

To understand how to check whether all functionality can be used via voice control.

Audiences affected
  • People with reduced motion
  • People who rely on voice contol to navigate and interact with controls
Common interactions using visible labels
  • Say: “Tap Voice control”
  • Say: “Tap Voice access”
  • Say: “Tap Email address”
  • Say: “Tap Subscription topic”
Common interactions using numbers
  • Say: “Show numbers”
  • Say: “Tap [number]
Test: Tool check
  1. Turn on voice control:
    • iOS: Settings → Accessibility → Voice Control → ensure Show Names (or Show Numbers) is on.
    • Android: Settings → Accessibility → Voice Access. (You can say “Show labels” or “Show numbers.”)
  2. Check: All controls are operable by voice alone (no touch needed).
  3. Check: Speaking the visible label reliably activates the correct element.
WCAG requirements
  • SC 2.5.3: Label in Name (Level A): The visual label for controls is a trigger for speech activation.
  • SC 4.1.2: Name, Role, Value (Level A): People using assistive technology understand all components.
  • SC 1.3.1: Info and Relationships (Level A): Information about content structure is available to more people.
