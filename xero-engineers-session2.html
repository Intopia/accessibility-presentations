Screen readers and forms

It is important to understand how screen readers operate, especially when interacting with form elements.

The virtual buffer

Browsers go through a series of steps when creating web documents:

  • Construct the DOM
  • Construct the CSSOM
  • Construct the Render tree
  • Layout the Render tree
  • Paint the Render tree on screen

At some point in this process, browsers also construct their version of the accessibility tree - a simplified version of the DOM.

Screen readers analyze the accessibility tree and create a linear version of this information.

This information is then stored in a local ‘virtual document’ or a ‘virtual buffer’.

As the user navigates through this virtual buffer using keystrokes, the focal position is referred to as a ‘virtual cursor’.

Two modes

The NVDA and JAWS screen readers have two different modes:

  • Browse mode
  • Forms mode (JAWS)/Focus mode (NVDA)

‘Browse mode’

‘Browse mode’ is used when navigating through the virtual buffer to read documents or web pages.

Some keystrokes can be used as navigation shortcuts in ‘Browse mode’. For example, the letter H will take users to the next heading.

‘Forms/Focus mode’

‘Forms/Focus mode’ is used when interacting with form fields that require user input.

When tabbing in ‘browse mode’ and reaching an element requiring keyboard input, JAWS and NVDA can automatically switch to ‘forms/focus mode’.

* Depending on user settings

When tabbing in ‘browse mode’ and reaching an element requiring keyboard input, JAWS and NVDA can automatically switch to ‘forms/focus mode’.

* Depending on user settings

Switching to ‘forms/focus mode’ mode will cause the screen reader to leave the virtual buffer.

Some screen reader users prefer to switch manually between these two modes, so that they have more control over the experience.

How do screen reader users navigate forms?

There are many ways these users may navigate through forms.

Method 1:

Reading line-by-line through the form in ‘Browse mode’ to understand the form’s purpose.

Method 2

Navigating via specific keystrokes (in JAWS and NVDA) to jump directly to form controls.

Key Task
F Next form field
B Next button
R (NVDA) or A (JAWS) Next radio button
X Next checkbox

After using the F key to navigate to a form field, users can use ENTER to switch into ‘Forms/Focus mode’ if they need to type data into the field.

Method 3

Navigating via a list of all form fields - e.g using the Elements List (NVDA and JAWS) or the Rotor (VoiceOver).

Method 4

Navigating via the TAB key, allowing the user to move to the next element that can receive focus - links, buttons, form fields.

‘Forms/Focus mode’ and form content

There is something to remember when users navigate through a form using F or TAB keys.

Any content within the form that cannot receive focus or is not programmatically associated with a form field will not be announced.

For this reason, any information relevant to a form field must be programmatically associated with that field.

  • Label text
  • Radio group and checkbox group headings
  • Form field instructions
  • Form field error messages
  • Required field notifications

How names and descriptions are announced

How are the accessible name and description announced in different screen readers?

Screen readers announce objects within the accessibility tree in the following order:

  • User value (if present)
  • Accessible name
  • State (Disabled, checked, unchecked etc.)
  • Role
  • Description
OSX / VoiceOver / Firefox and Safari
  • Accessible name: ‘Address’
  • State: ‘Required’
  • Role: ‘Edit text’
  • [Pause]
  • Description: ‘Include full street address’
OSX / VoiceOver / Chrome
  • Accessible name: ‘Address’
  • State: ‘Required’
  • Role: ‘Edit text’
  • [Pause]
  • Description: ‘More content available’
Windows / NVDA / Chrome
  • Accessible name: ‘Address’
  • Role: ‘Edit’
  • Description: ‘Include full street address’
  • Additional instructions: ‘Blank’
Windows / JAWS / Chrome
  • Accessible name: ‘Address’
  • Role: ‘Edit’
  • Description: ‘Include full street address’
  • Additional instructions: ‘Type in text’

Let’s review some forms

Some form examples
  • Contacts > New Contact
  • Payroll > New employee
  • Any other form you'd like to review
1. Do all form fields have accessible names?
  • Do they use matching for and id values?
  • Do they rely on lesser methods, such as placeholders?
  • Test cases
2. Do you use fieldsets for radio and checkbox groups?
  • Are fieldsets and legends used?
  • Is there any misuse of the button group?
3. Are your instructions programmatically associated?
  • Do your instructions generate an accessible description?
4. Are your required fields visually identifiable?
  • Do required fields have some sort of visual indicators?
    • Some sort of symbol
    • Required fields marked with (‘required’)
    • Optional fields marked with (‘optional’)
  • If required fields are not all identified, is there an overall message?
5. Are your required fields programmatically identifiable?
  • Are required fields defined as required in the accessibility tree?
6. Are your error fields programmatically identifiable?
  • Do errors generate an accessible description?
  • Are invalid fields defined as invalid in the accessibility tree?
  • Are error states triggered via onBlur or when submitted?
7. What happens when a user submits a form with errors?
  • Where does focus go?
  • Are users informed:
    • That there are errors?
    • The number of errors?
    • How to navigate to each error?
