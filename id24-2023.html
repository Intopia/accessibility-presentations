What makes an accessible date picker? Is it even possible?

Inclusive Design 24

A decision tree?

Let’s assume your team wants to use a date picker as part of your website or design system.

Where would you start?

We could start with a fundamental user experience question:

What is the best method of allowing users to enter a date quickly and easily?

Is the date picker pattern the best solution for your key audiences?

Sometimes, people jump straight into a solution without asking these critical questions.

And we often bring our own biasses.

For example, I used to think that date pickers were the work of Satan.

But, how might two different people interact with a date picker?

Andrew
  • A person who is blind

People

You can start by looking at different groups of people and their needs.

This means you can focus on one user type and explore how they may need to interact with the component.

These groups of people could include:

  1. People with some sort of cognitive impairment or neurodivergence.
  2. People with some sort of colour vision deficiency.
  3. People with low vision.
  4. People with limited mobility.
  5. People with limited or no sight.

1. People with some sort of cognitive impairment or neurodivergence

We need to make sure:
  • The purpose of the component is clear.
  • Any additional instructions are clear.

2. People with some sort of colour vision deficiency

We need to make sure:
  • The component has sufficient colour contrast.

3. People with low vision

We need to make sure:
  • The text and layout are understandable when scaled.
  • The component works when viewed in low contrast.

4. People with limited mobility

We need to make sure:
  • The component is accessible for keyboard-only users.
  • The component is accessible for voice control users.
  • Focus states are visible.
  • Focus order is intuitive.

5. People with limited or no sight

We need to make sure:
  • The component, and any relevant interactive elements have an accessible names.
  • All possible states and values are defined.
  • Any dynamic content is announced at the appropriate time.

We can use this information to create a cohesive set of accessibility unit tests.

Accessibility unit testing

“Accessibility unit testing” is a process where the smallest parts of a component are individually and independently evaluated.

Accessibility unit tests are often based on two concepts:

1. How all aspects of the component are presented to users - either on screen or via assistive technologies.

2. How users should be able to interact with the component to complete all relevant tasks.

These tests must include all possible happy and unhappy paths associated with the component.

And there are different methods that people use to achieve these interactions - mouse, keyboard, assistive technologies etc.

Each possible task needs to be defined as a test.

And every test must be written as a statement where the only outcome is either a pass or a fail.

Each test must be repeatable so that it can be re-tested at a later date.

An example

We can now create a chart of possible acessibility unit tests for each of our different types of people.

We will start with “People with some sort of cognitive impairment or neurodivergence”.

If the component has issues for these users, then there are fundamental UX issues and the component should re re-designed.

We will then look at design-related concepts for:

  • People with some sort of colour vision deficiency
  • People with low vision

And finally, we will look at more technical concepts for:

  • People with limited mobility
  • People with limited or no sight

We should run keyboard-only tests before screen reader tests, as screen reader users need the same basic functionality with additional criteria.
