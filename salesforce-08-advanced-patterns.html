Advanced patterns

In this video, we will explore some common advanced patterns.

These include carousels, tab panels, accordions and sticky components.

Carousels

Carousels create many potential accessibility barriers.

A carousel should have the following:

Visible components with appropriate target size.

A visible focus indicator.

A progress indicator.

If the carousel auto-rotates, a ‘pause’ button should be provided.

Accessibility considerations for carousels include:

Can a screen reader tell which slide is currently in focus, and hear all relevant content?

Is the focus order logical?

Can a keyboard user reach all content?

Can a touch screen user reach all content?

Can the movement of the carousel be paused?

Keyboard support:

Use the <button> element for interactive components.

Semantic support:

Wrap the carousel in a <section> element with an aria-label so that it can be identified as a distinct section of the page.

Ensure each slide has appropriate alt text or aria-label to identify which slide it is, as well as its content if needed.

Apply aria-hidden to any slides not currently in focus, so they are not announced.

If the carousel auto-rotates, apply aria-live to ensure dynamic changes are announced.

Prevent focus from going to any off-screen slides.

Check out this automatically scrolling carousel on the Agentforce page.

When focus enters the carousel, the tiles are individually focusable, but movement doesn’t pause.

This means the focus indicator is off screen and impossible to follow.

Example: https://www.salesforce.com/au/agentforce/

On this Field Service Management page, the carousel is easier to use.

Although the scrolling arrows are not keyboard accessible, the carousel tiles move with keyboard focus, ensuring the focused tile remains on screen.

Note how a screen reader announces each tile as belonging to a group of seven, making the size of the carousel clear.

Example: https://www.salesforce.com/service/field-service-management/

Tab panels

Over the years, developers have used a range of different methods for the tab mechanisms, including:

  • Headings
  • Links
  • Buttons

These days, buttons are the preferred method.

So, each tab panel should have a <button> element to reveal the tab.

Accessibility considerations for tab panels include:

Is the focus order logical?

Can a keyboard user reach all content?

Can a screen reader hear all relevant content when opened?

Keyboard support:

In the past, the most common method of navigating between tabs was via TAB keystrokes.

However, the most common method used today is using left and right ARROW keys.

Navigation from the tablist the the tabpanels should be done by TAB keystrokes

For example, on this Salesforce marketing page, the in-page tabs are accessible by using the keyboard ARROW keys.

Once a tab is selected with ENTER, focus can enter the tab panel.

Example: https://www.salesforce.com/marketing/

Semantic support:

The tablist role identifies the element that serves as the parent container for a set of tabs.

<div role="tablist">
  <button role="tab">Panel name</button>
</div>
<div role="tabpanel">
  Panel information
</div>

Each <button> inside the tablist should be given a role of tab.

The tab role indicates an interactive element that, when activated, displays its associated tabpanel.

<div role="tablist">
  <button role="tab">Panel name</button>
</div>
<div role="tabpanel">
  Panel information
</div>

And finally, each panel that is revealed or hidden below should have a role of tabpanel.

The tabpanel is a container for the resources of layered content associated with a tab.

<div role="tablist">
  <button role="tab">Panel name</button>
</div>
<div role="tabpanel">
  Panel information
</div>

Accordions

An accordion should have the following:

A header that explains what’s inside.

A panel which contains the related content.

A visual indicator which shows if the panel is expanded or collapsed.

Accessibility considerations for accordions include the following:

Is it keyboard accessible?

Is it clear to a screen reader whether the panel is expanded or collapsed?

Is the visual indicator in close proximity to the header text (preferably on the left)?

Keyboard support:

All focusable elements can be reached by TAB.

When the header element receives focus, use ENTER or SPACEBAR to open it.

Semantic support:

The trigger should be a <button> element.

These buttons are often wrapping in a heading <hn> element.

Panel content should be in a <div>.

The aria-expanded attribute should be applied as true when the accordion is expanded and false when it is closed.

This will indicate the state of the panel to assistive technologies.

Have a look at this accordion on the Salesforce analytics page.

Each panel is accessible by keyboard.

A screen reader announces when a panel is expanded or collapsed.

Example: https://www.salesforce.com/analytics/

Sticky components

Accessibility considerations for sticky components include:

Can the sticky component receive focus, and is the focus order logical on the page?

Does the sticky component prevent other elements from receiving focus, or obscure the focus indicator?

Where in the page order is the sticky component announced?

When a page is reflowed or viewed via a small viewport, does the sticky component obscure page content?

Keyboard support:

All focusable elements can be reached by TAB.

Support:

For example, on this Dreamforce page, we have a sticky countdown clock and a sticky Agentforce button.

When we test the page at 400%, the sticky content becomes large, and obscures parts of the page, making it difficult to read.

There is no way to remove this content.

This sticky component is also a link, which means it needs to receive keyboard focus.

It currently receives focus after the main banner, but before the component actually appears visually.

It would be very difficult for a user to understand where focus has gone, and access this link.

Example: https://www.salesforce.com/dreamforce/

We test the WCAG Reflow criterion at 1280 x 1040, at up to 400% zoom.

But people may be accessing your site at many different sizes and magnifications.

It's a good idea to check that it doesn't break under other conditions.

Conclusion

Following these suggestions will make your advanced patterns easy to use for more people.
