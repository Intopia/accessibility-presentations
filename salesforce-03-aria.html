Understanding ARIA

What is ARIA?

ARIA stands for Accessible Rich Internet Applications.

It helps make websites and web apps more accessible to people using assistive technologies.

ARIA is a set of attributes that add roles, states, and properties to the accessibility tree, helping assistive technologies interpret complex or custom UI elements.

The current version is WAI-ARIA 1.2.

What problems is ARIA trying to solve?

ARIA aims to solve two key problems in accessibility:

1. Dynamic content updates

When content changes via JavaScript, assistive technologies often aren’t notified unless ARIA is used to communicate those changes.

2. Custom UI components

Many modern interfaces use non-native elements (like custom dropdowns or modals) that don’t expose the right semantics to assistive technologies.

ARIA helps fill in the missing information.

ARIA is like a polyfill between the HTML we have now and the HTML we wish we had — or may soon have!

In fact, our aim is to gradually avoid or even remove ARIA as HTML and browser support improves over time.

Roles, states and properties

ARIA attributes are broken down into three categories: roles, properties and states.

ARIA roles

ARIA roles can be used to add or change the semantic meaning of HTML elements in the accessibility tree.

ARIA roles are HTML attributes written as:

role="[role name]"

Let’s look at two examples of how we could add a role to an element.

Using role="button"

We can add role="button" to the <div>.

<div role="button"></div>

Example: https://intopia.github.io/exercise/salesforce-testing-role-button.html

If we look at the first element in the accessibility tree, we can see that it has a role of generic.

If we look at the second element in the accessibility tree, we can see that it has a role of button.

We should avoid adding ARIA roles unnecessarily, and use native HTML elements like <button> whenever possible.

Native elements are better supported and come with built-in keyboard and accessibility features.

Using role="combobox"

We can add role="combobox" to an <input>.

This will change the element’s role from textbox to combobox in the accessibility tree. 

<input role="combobox">

This new role provides additional context to assistive technologies.

It will help them understand that the element behaves like a dropdown or autocomplete widget.

ARIA properties

ARIA properties provide additional information about HTML elements to the accessibility tree.

ARIA properties are HTML attributes written as:

aria-[property]="[value]"

Let’s look at two examples of how ARIA properties could be used.

1. Using aria-label

This <button> contains an "X" symbol to indicate that it will close or dismiss something.

However, it has no accessible name.

We can add an aria-label="Dismiss" to the <button>, so that we can give it a meaningful accessible name.

<button aria-label="Dismiss">X</button>

Example: https://intopia.github.io/exercise/salesforce-testing-aria-label.html

If we check the element’s name in the accessibility tree, we can see that it now has a name of “Dismiss.”

If we listen to the button announced by VoiceOver, we can hear the word “Dismiss” announced.

DEMO

This attribute is useful when you need to give an element a name or enhance it for assistive technologies - especially when there’s no visible label.

2. Using aria-live

We can add aria-live="polite" to elements.

In this case, the aria-live attribute is applied to the <div>.

<div aria-live="polite">
  Hello, I am some dynamically inserted content!
</div>

This tells assistive technologies that the element is a live region — a place where content might change without the user interacting.

The polite value means screen readers will wait until the user is idle before announcing the change.

3. Using aria-describedby

We can add aria-describedby="info" to the <input> and link it to a description elsewhere on the page.

It’s helpful for providing hints, guidance, or error messages that should be read after the element’s accessible name.

In this case, we are linking the instructions to the <input>.

<label for="username">Username</label>

<span id="info">Your username must be at
least 8 characters long.</span>

<input id="username" aria-describedby="info">

Example: https://intopia.github.io/exercise/salesforce-testing-aria-describedby.html

If I check the element’s name in the accessibility tree, we can see that it now has a description of “Your username must be at least 8 characters long”.

If we listen to the input announced by VoiceOver, we can hear the label announced - “Username” as well as the instructions - “Your username must be at least 8 characters long”.

ARIA states

ARIA states are used to inform users about the current status or condition of elements in the accessibility tree.

Unlike ARIA properties that describe permanent characteristics, states are dynamic - they change as the user interacts with the page.

These changes are usually managed with JavaScript.

ARIA states are HTML attributes written as:

aria-[state]="[value]"

Let’s look at two examples of how ARIA states could be used.

1. Using aria-expanded

We can add aria-expanded="true" to the <button>.

<button aria-expanded="true"></button>

This attribute is typically added or updated when the button is triggered.

For example, when it opens a menu or a collapsible panel.

It informs assistive technology users that the element controlled by the button is currently expanded.

2. Using aria-invalid

We can add aria-invalid="true" to the <input>.

<label for="password">Password</label>
<input id="password" aria-invalid="true">

This state is usually applied some time after the user enters invalid data in the input.

It notifies assistive technologies that the field has an error and requires correction.

Example: https://intopia.github.io/exercise/salesforce-testing-aria-invalid.html

If we check the element in the accessibility tree, we can see that it now has a a state of “Invalid user entry: true”.

If we listen to the input announced by VoiceOver, we can hear the label announced - “Name” as well as the state - “Invalid”.

ARIA can’t do everything!

It’s important to understand what ARIA does not do.

ARIA is not a fix-all tool; it only enhances accessibility by communicating semantic information to assistive technologies.

Specifically, ARIA does not do any of the following:

Modify an element’s visual appearance.

ARIA won’t change how something looks on screen. You still need CSS for styling.

Modify an element’s behavior.

ARIA doesn’t add or change how elements behave; that’s controlled by your HTML, JavaScript, or other scripting.

Add focusability to elements.

If an element isn’t natively focusable, ARIA won’t make it focusable. You need to use proper HTML or tabindex for keyboard focus.

Add keyboard functionality to elements.

ARIA doesn’t automatically enable keyboard interactions. You must implement keyboard event handlers yourself.

Support for ARIA

How can we determine if ARIA is supported by each of the different browsers and assistive technologies?

The A11ySupport.io site is a great resource that documents support for individual ARIA attributes across major browsers.

The HTML test cases site show how various HTML elements and ARIA attributes behave in different screen reader and browser combinations.

ARIA support can vary depending on browser and assistive technology versions.

For this reason, it’s always best to test your own implementations in the environments your users rely on.

Conclusion

ARIA can be a very powerful and effective tool to help improve accessibility, as long as it used well.
