Screen readers

It is important to understand how screen readers operate, especially when interacting with form elements.

Browsers go through a series of steps when rendering web documents, starting with the creation of the Document Object Model.

The Document Object Model (DOM) defines the logical structure of documents and the way a document can be accessed and manipulated.

Browsers also construct an accessibility tree - a simplified version of the DOM.

The accessibility tree consists of information about specific HTML elements, including their:

  • Name (e.g. ‘Submit’, ‘Full name’)
  • Role (e.g. button, link, textbox)
  • State (e.g. checked, expanded, disabled)
  • Value (e.g. input from the user)

Screen readers analyze the accessibility tree and create a linear version of this information.

This information is then stored locally as a ‘virtual document’.

Screen reader users can navigate this virtual document using a range of keystrokes.

For example, JAWS and NVDA use the following keys to find and navigate to headings:

Key/function Purpose
H Jump to next heading
SHIFT + H Jump to previous heading
1, 2, 3, 4, 5, 6 Jump to next heading of that type
SHIFT + 1, 2, 3, 4, 5, 6 Jump to previous heading of that type
Elements list popup View all headings

As screen reader users navigate through this virtual document, the focal position is referred to as a ‘virtual cursor’.

Two modes

The NVDA and JAWS screen readers have two different modes:

  • Browse mode
  • Forms mode (JAWS)/Focus mode (NVDA)

‘Browse mode’

‘Browse mode’ is used when navigating through the virtual document to read the content.

‘Forms/Focus mode’

‘Forms/Focus mode’ is used when interacting with form fields that require user input.

When tabbing in ‘browse mode’ and reaching an element requiring keyboard input, JAWS and NVDA can automatically switch to ‘forms/focus mode’.

* Depending on user settings

When tabbing in ‘browse mode’ and reaching an element requiring keyboard input, JAWS and NVDA can automatically switch to ‘forms/focus mode’.

* Depending on user settings

Switching to ‘forms/focus mode’ mode will cause the screen reader to leave the virtual document.

Some screen reader users prefer to switch manually between these two modes, so that they have more control over the experience.

How do screen reader users navigate forms?

There are many ways these users may navigate through forms.

Method 1:

Reading line-by-line through the form in ‘Browse mode’ to understand the form’s purpose.

Method 2

Navigating via specific keystrokes (in JAWS and NVDA) to jump directly to form controls.

Key Task
F Next form field
B Next button
R (NVDA) or A (JAWS) Next radio button
X Next checkbox

After using the F key to navigate to a form field, users can use ENTER to switch into ‘Forms/Focus mode’ if they need to type data into the field.

Method 3

Navigating via a list of all form fields - e.g using the Elements List (NVDA and JAWS) or the Rotor (VoiceOver).

Method 4

Navigating via the TAB key, allowing the user to move to the next element that can receive focus - links, buttons, form fields.

‘Forms/Focus mode’ and form content

There is something to remember when users navigate through a form using F or TAB keys.

Any content within the form that cannot receive focus or is not programmatically associated with a form field will not be announced.

For this reason, any information relevant to a form field must be programmatically associated with that field.

  • Label text
  • Radio group and checkbox group headings
  • Form field instructions
  • Form field error messages
  • Required field notifications
