A quick intro to Git and GitHub

1. Why Use Version Control?

  • Avoids messy file naming: file-v3-final.doc.
  • Maintains a complete history of every change.
  • Enables collaboration without overwriting work.
  • Makes bug tracking and rollback easy.
  • Encourages experimentation with branching.

2. What Is Git?

Git is a version control tool that helps developers save different versions of their work over time.

Git is widely used in modern software development to keep track of code changes, collaborate with others, and avoid losing work.

Git works on your local machine and doesn’t require an internet connection for day-to-day tasks like saving changes or creating branches.

Key features:

  • Stores complete snapshots of a project.
  • Works locally on your machine.
  • Every user has the full history.
  • Enables safe experimentation.
  • Efficient and fast.
  • Flexible and widely supported.

3. Why Developers Use Git

  • Maintains a detailed history of changes.
  • Makes it easy to revert to previous versions.
  • Allows multiple developers to work safely on the same project.

4. Core Git Concepts

4.1 Basic actions

  • Repository: A project folder tracked by Git.
  • Pull: Get changes from a shared project and bring them into your local version.
  • Push: Send your saved changes back to the shared project.
  • Commit: A saved version of your changes with a message.

4.2 Branching and merging

  • Branch: A separate copy of your code to test or add features.
  • Merge: Combine changes from one branch into another.

4.3. Copying projects

  • Clone: Copy a Git project from another location (often from GitHub).

5. What Is GitHub?

GitHub is a cloud-based platform for hosting Git repositories.

It provides a central place for the code — one source of truth.

GitHub makes it easier to collaborate, especially on large or distributed teams.

It tracks who changed what, when, and why.

Key features:

  • Supports both public and private projects.
  • Enables global collaboration.
  • Provides detailed history and tracking.
  • Visualises contributions and activity.
  • Offers tools for managing projects, not just code.
  • Works as a central source of truth for teams.

6. Common GitHub Features

6.1 Pull Requests:

  • Propose code changes.
  • Review, comment, and approve before merging.

6.2. Issues:

  • Report bugs or suggest features.
  • Assign, label, and track progress.

6.3. Discussions & Wikis:

  • For team Q&A and documentation

7. Common tools

Developers can use Git in many ways: through the terminal, inside their code editor, or through dedicated apps.

7.1. Command Line / Terminal

  • Most common and powerful way to use Git.
  • Commands like git status, git add, git commit, etc.
  • Works on macOS, Windows (via Git Bash), and Linux.

7.2. Integrated Development Environments (IDEs) and Editors

VS Code: A popular code editor with built-in Git features like commits, branches, and file comparisons.

JetBrains IDEs: Powerful Git integration with visual tools for commits, logs, branches, and merges.

Eclipse: Includes Git support via the EGit plugin.

Xcode (macOS): Offers built-in Git source control features for Apple developers.

7.3. Standalone Git GUIs (Graphical User Interfaces)

GitHub Desktop: Simple tool for cloning, committing, branching, and syncing GitHub projects.

Sourcetree (by Atlassian): Visual Git and Mercurial tool, great for managing branches and resolving conflicts.

GitKraken: Cross-platform Git client with a visual commit history and drag-and-drop merges.

Tower: Professional-grade Git client with visual features and GitFlow support.

7.4. Online Platforms

These platforms are web-based and allow users to view and sometimes edit code, track changes, and collaborate on development.

GitHub: View code, commits, branches, pull requests, and use in-browser editing.

GitLab: Self-hosted or cloud-based Git platform, popular for DevOps pipelines.

Bitbucket: Atlassian’s Git service with Jira integration.

8. FAQ

8.1. Do I need to know Git if I’m not a developer?

Not necessarily. But understanding how Git and GitHub work can help you collaborate better with developers — especially if you're involved in review or testing.

8.2. What’s the difference between Git and GitHub again?

Git is the tool used to save versions of code on your computer. GitHub is an online platform where teams share that code and work together.

8.3. Can I break something by looking at a GitHub repository?

No. Just viewing a repository or reading code won’t cause any changes. You need permission to make edits or suggestions.

8.4. What is a “commit message”?

It’s a short note a developer writes when they save a change, explaining what was updated or why.

8.5. Why are there so many branches?

Branches let developers try out new features or fixes without affecting the main project. It keeps things safe and organised.

8.6. Can GitHub be used for things other than code?

Yes! Some teams use it to manage documentation, design files, or even plain text projects — especially when they want version history or collaboration.

8.7. Is GitHub only for open-source projects?

Not at all. Many private teams and companies use GitHub to manage internal projects that the public never sees.

8.8. How do people review each other’s work?

They usually create a pull request, which shows the proposed changes. Others can then comment, request improvements, or approve the update before it goes live.

8.9. What does “pull” mean?

“Pull” means getting the latest version of a project from GitHub so that your local copy is up to date.

8.10. Can I try GitHub without installing anything?

Yes! You can create a GitHub account and explore public repositories or even edit files in your browser without needing to install Git.
