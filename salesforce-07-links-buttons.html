Accessible links and buttons

Links are a navigation tool for users moving between or within pages.

Use a link to connect two resources.

Buttons should trigger an action on a page.

Use a button to enable a user to perform a programmable action, such as submitting a form or opening a dialog.

Both links and buttons need to be designed and implemented in a clear and accessible manner.

This makes it easy for all users to take the action they require.

Well-implemented links and buttons can make interactions simpler and more efficient for people with low or no vision, reduced mobility, or varied cognition.

Visual appearance

When designing both links and buttons, there are several accessibility considerations to take into account.

Colour is an important factor.

Links must have a contrast ratio of 4.5:1 (for small text) or 3:1 (for large text) against their background.

Active buttons (without a text label) must have a contrast ratio of 3:1 against their background.

All links and buttons must show a visual focus indicator when in focus.

This focus indicator must also have appropriate colour contrast (3:1).

Example: https://intopia.github.io/exercise/salesforce-testing-link-button.html

Notes: Notice the focus styles as I focus on the link and button.

All links and buttons must be visually distinguishable from surrounding non-link text.

That means something other than colour to indicate they’re interactive.

This could be an underline for a link, or a chevron for a button.

If possible, combine elements going to the same location into the same link.

For example, on this Salesforce page, each tile has three separate link components that all go to the same place.

Bundling these together provides a clearer and more efficient user experience.

Problematic example: https://www.salesforce.com/communications/sales-software/

Notes: Under ‘Learn new skills…’ each tile has three separate links to the same place.

Visible text labels

A visible text label is a label that is presented on-screen as text.

Visible labels can include the contents of links and buttons.

For a link that uses the link text Intopia Home, “Intopia Home” is the visible label.

For a button that reads “Save”, this is the visible label.

Each element should have a unique, accessible label that distinguishes it from other elements on the page.

That means two things:

1. Buttons that perform the same action, or links that go to the same location, should have the same or similar names.

2. Buttons that perform different actions, or links that go to different locations, should have different names.

Links that open in a new window or tab should have visual and programmatic indicators, so users know before clicking on them.

Links that point to a file or destination in a non-HTML format (e.g. MS Word, PDF, plain text, etc.) should indicate the type of file or destination.

Ideally, these links should also indicate the size of the file.

It should be clear to a user what a link or button will do if it is activated.

Check out this ‘Become a sponsor’ link. When activated, it opens a blank email.

This action is not made clear by the link text, which means a user may not understand what has happened.

Problematic example: ‘Become a sponsor’ near bottom of https://www.salesforce.com/dreamforce/

Images

Images that function as links should be given alt text that indicates where the link will go.

For example, the logos on this page have the alt text format of “company name, consultant overview”.

This makes it clear where each image link is going.

The aria-label attribute

Sometimes, we need to add extra context to how a link or button will be interpreted by assistive technologies.

We can do this by using the aria-label attribute.

An aria-label will provide an accessible, alternative label for assistive technologies.

For example, it can be used to distinguish between links with the same link text (e.g. “Read more”).

It can also add extra information about how a button works.

An aria-label is applied directly to the element itself.

<button aria-label="Save my details">
  Save
</button>

An aria-label will not affect what is presented as the visible text label.

Visible text labels and accessible names should always match at least partially.

Example: https://intopia.github.io/exercise/salesforce-testing-aria-label.html

Note: If we listen to the button announced by VoiceOver, we can hear the word “Dismiss” announced.

Here is an example where the accessible name almost matches the visible text label.

<button aria-label="Save current work">
  Save
</button>

In the following example, the aria-label attribute has been used to give the button an accessible name that is entirely different to the visible text label.

<button aria-label="Add to cart">
  Save
</button>

Example: https://intopia.github.io/exercise/salesforce-testing-aria-label-bad.html

Note: If we listen to the button announced by VoiceOver, we can hear the words "Add to cart" announced.

A mismatch between the visible text label and the accessible name could confuse some types of users.

It will also result in barriers for users of voice recognition technology.

Since aria-label content is only presented to assistive technologies, consider making important information visible for all users.

Use an aria-label when:

1. Sighted-users are getting information that non-sighted users are not.

2. There is no better way to convey this information through a visible label.

Conclusion

Following these guidelines will make links and buttons easy to use for more people.
