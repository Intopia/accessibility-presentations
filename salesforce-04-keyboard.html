Testing for keyboard accessibility

Introduction

Some of our users primarily operate their devices with a keyboard or keyboard alternative (like a switch device).

This might include people with limited physical ability who cannot use a mouse, or people who are blind and use a screen reader.

It can also include power users, and people experiencing technical issues.

We need to ensure that all functionality is available using standard keystrokes for all the different types of people who may need it.

Keyboard use

Most native elements already have built-in keyboard operability.

“Native” means using HTML elements for their intended semantic purpose.

Examples of native elements could include:

  • Links
  • Buttons
  • Inputs
  • Select (dropdown) menus
  • Textareas

On the other hand, ‘Non-native’ implies using HTML elements in a manner that deviates from their intended purpose.

Examples of non-native elements could include:

  • Modal windows
  • Date pickers
  • Autocomplete dropdown menus
  • Accordions
  • Complex navigation menus

Keyboard access

Every page should be accessible via the keyboard - without having to use the mouse.

You should be able to use the following keystrokes to navigate and interact with page elements.

  • Use Tab to move forward.
  • Use Shift + Tab to move back.
  • Use Enter to trigger a link.
  • Use Enter or Spacebar to trigger a button.
  • Use Escape to close modals and dropdowns.
  • Use Arrow keys to navigate through menus, radio button groups and dropdowns.

For example, on the Salesforce news page, we can use the TAB key to access each focusable element.

We can use Enter to trigger a link.

We can use Spacebar to open a dropdown menu.

And Arrow or Tab keys to access each item inside the menu.

We can use Escape to move out of the menu.

Example to use: https://www.salesforce.com/news/

Have a go on a page of your choice.

You should see your focus indicator moving through the page.

Take note of the following:

How easy is it to see?

How logical is the order in which it moves?

How many times do you have to key to get to the content you want?

We’ll look more into these ideas as we go.

Focus order

There are three related concepts when it comes to a page order:

1. Source order, or the order in which HTML markup is written.

2. Visual reading order, or the order in which a sighted user would naturally read the page contents.

3. And keyboard focus order.

Keyboard focus order is the order of interactive page elements, such as links and form controls, when accessed by the keyboard.

We can determine the keyboard focus order by tabbing through elements on a web page.

Keyboard-focusable items should receive focus in a sequential manner that is intuitive and predictable.

We must ensure that people using keystrokes to navigate can always predict where focus will go as they move through the page.

Let’s check the focus order on the Salesforce pricing page.

Focus order is managed in a logical way on this page.

Every component is reachable by keyboard in the order we would expect.

This popup chat window floats independently of the page.

That means it correctly appears as the very last item in the page’s focus order.

Interestingly, when we reach the page footer, our focus goes below the chat window into the footer region, before we reach it.

While this is unavoidable, it could be jarring for some users, who might assume the window doesn’t receive focus at all.

Example to use: https://www.salesforce.com/sales/pricing/

As we go TAB through this page, you will see that the chat window is the last item on the page in focus order - even though it appears above the final footer information.

This could be confusing for people who rely on the keyboard or a keyboard alternative to navigate.

Focus indicators

A focus indicator lets a person know which element on the screen has focus.

A good focus indicator:

1. Is consistent and clear – it should be the same on all pages across a product.

2. Does not rely on colour alone – it should be identifiable by more than just a colour change (for example a border, underline, or chevron).

3. Has good contrast – it should have a contrast ratio of at least 3:1 against adjacent colours.

Let’s have a look at the focus indicator on the Salesforce news page.

As we begin to tab through the page, the focus indicator is clear and easy to see.

It has appropriate weight and colour contrast.

However, when focused on the carousel navigation bar, the focus indicator shrinks.

It is very difficult to see.

When we reach the ‘View all episodes’ component in the CIO Conversations section, the focus indicator goes missing altogether.

We still have focus, and can activate the link, but we cannot see where we are.

Use: https://www.salesforce.com/news

Focus management

When content dynamically changes, we want to keep the focus logical for both sighted and non-sighted users.

For example, when a modal window is triggered, we should set focus to one of the following:

  • The modal window container,
  • The first heading inside the modal window,
  • Or the first interactive element inside the modal window.

The best solution may depend on the circumstances.

For example, when users try to submit simple forms with errors, we should send focus to the first field in error.

For more complex forms, we should send focus to an error message above the form.

Single page apps

Single-page web applications change the content on the page as you interact with it.

They do this without needing to load a whole new page each time.

Instead, it updates the current page with new information, making everything feel faster and smoother.

One common problem with these pages is that developers may not send focus to an intuitive location.

This could be confusing for keyboard and screen reader users as they may not understand what has happened.

Ideally, we want focus to be sent to the main page heading each time the page content updates.

Check out this demo of a single page app.

As we submit each step, notice that focus moves to the page heading.

This allows assistive technology users to orientate themselves as the page refreshes.

Example: https://intopia.github.io/exercise/salesforce-testing-focus-management.html

There are also times when we may want to avoid sending focus to a specific location.

An example could be a simple toast notification that pops up for 5 seconds and tells us our information has been saved.

In these circumstances, we do not need to shift focus at all.

Another example occurs on the Salesforce marketing page.

These cards each contain three interactive elements: the image, the header, and the call to action.

To reduce keystrokes, the image has been removed from the focus order, as it goes to the same location as the other links.

Example: https://www.salesforce.com/marketing/

Regardless of the scenario, our focus management system is designed to be adaptable, ensuring an efficient and logical experience for all users.

Conclusion

Testing keyboard is a fairly straightforward process.

We’re looking for behaviour that is logical, predictable, intuitive, and efficient.

Remember, just because something can be accessed, doesn’t make it easy to access.

If a keyboard user cannot easily guess the page order, or must use 85 tabs to reach the submit button, we are not providing a usable experience.
