Link text

Links must tell you where you’re going.

The most fundamental thing about links is that the text tells you where you’ll go if you follow it.

Think about click bait links that sound intriguing, but when you click them, you don’t end up where you expected. This is not only annoying to your readers but erodes their trust in your site.

I also regularly see links that, rather than take you directly to the content in question, will take you to the site’s home page, in case the site’s structure changes in future.

It’s better to have the links go directly, where possible, straight to the relevant information and deal with possible broken links when you need to.

Link text must be independent.

Link text must not rely on context from headings or nearby images or text. While this approach might be okay for visual readers, it doesn’t work well for anyone relying on a machine reading code, like a screen reader.

And then the first of the Tell me more and Get a quote links. You can see why our screen readers don’t know that they’re referring to Home insurance or Car insurance.

Links with the same name go to the same place.

As you saw, all the "get quote" links actually went to different pages because they related to different insurance types. Conversely links with different names should go to different places. don’t call one link “Phone us” and another link “Email us” if they’re both going to the contact us page.

Links are easy to see.

We mentioned this in the last session, but you need to make sure that you can spot links amidst body text. While this isn’t something you may be able to change, you can point out to your design partners.

The other thing to note here is consistent use of link styles. You don’t have to stick to just underlining links in every instance. But having 4 or 5 different link styles on the same page goes against the principle of consistency.

Writing good link text

Put important information first.

Just like we discussed earlier, it’s important to ‘frontload’ your keywords in links just as it is in writing or alt text.

Keep links short, efficient yet descriptive.

Don’t rely on context for the link to make sense.

As we saw before with the get a quote links, relying on a visual heading to provide context for a link is not accessible.

Download links should contain file type and size information.

This lets people decide if they’ve got room on their device, Internet bandwidth or data, or even the necessary software before they download anything.

Similarly links that open to multimedia should say so. For example your link could be ‘watch our brewing process video’. Or just ‘brewing process [video]’.

Other link considerations

Link placement

Reduce mid-sentence links, where possible, as they can be distracting.

In the last module, we talked about scanning behaviours, especially for people who use screen magnification windows.

Research shows that mid-sentence links can be distracting. They can slow down readers who scan, as they need to stop and read text on both sides of the embedded link. They may cause readability challenges for autistic people.

See Readability Guidelines for the research about this.

This means if it's possible without rewriting content in an unnatural way, a link should go either at the beginning or at the end of sentences.

Having your links aligned to the left-hand side, or the beginning of sentences, makes them easier to spot.

It’s up to you how you want to style them: maybe you like putting links under their related paragraph. Or perhaps you prefer links in bulleted lists. Just be aware that content on the right-hand side of the page may be missed.

Opening links in new windows or tabs

Flag links that open in a new window or browser tab.

The acknowledged best practice always is to give people choice. Forcing a link to open in a new window or tab removes the choice. There are some instances when it’s better to open the link in a new window or tab though. For example if you’re in a multistep process, like filling in a form, a link that opens in the same window will mean you have to start all over again. Similarly if you are logged into something and you need to link to information outside that environment it’s better to open the link that you window rather than having it to log back in again later.

If you use an icon, make sure it has alt text that explains what it is.

Avoid adding titles to links.

Generally the only text you need on a link is the link text itself, or alt text if the link is an image. Sometimes people add extra text descriptions using the title attribute. Generally you will only see these when you hover over a link and a little tooltip appears. So keyboard or touch device users will be left out.

Helping screen reader users.

We’ve mentioned throughout the section are number of things that help screen reader users. Biggest of these is to make sure your links are meaningful by themselves and don’t need any additional context. However in situations where you may be unable to improve your link text there are ways you can add extra text for screen readers. Your developers can add extra code to accommodate visually hidden words.

Part 2: Images

There is a lot of things to think about when it comes to accessible images. You need to decide firstly what type of image you’re dealing with and then secondly, if the image isn’t decorative you need to think about alternative text. This is often known as alt text. We’ll be talking about this more in the section.

Image types

Let’s look at the four different types of images.

Image type 1: Decorative images
  • Could be removed without affecting the overall story
  • Should be ignored by assistive technologies
Image type 2: Informative images
  • Provide unique information
  • Require a text alternative that conveys the meaning
Image type 3: Functional images
  • Image used as links or buttons
  • Should include a text alternative that describes:
    • The function of the link (where it will take you)
    • The function of the button (what it will do)
Image type 4: Complex images
  • Complex images such as graphs or infographics
  • Should include:
    • a short text alternative
    • an detailed text description

To alt or not

<img src="image.png" alt="Alt text">

Decorative images need the alt attribute with no value.

It’s important to note that decorative images still need an alt attribute. They just don’t have any alt text. Make sure your content management system can support an empty or null alt attribute. Otherwise they get ignored by screen readers altogether.

<img src="images/cat.jpg" alt="">

Context and alt text

Alt-text depends on the context of the image on a page. Not the image itself.

Alt-text with no page context:

‘A nearly empty Harvard Stadium.’

Here we have a image.

With no context I would probably put the alt text as something like "A nearly empty Harvard Stadium". To me, the stadium has focus here, possibly all the empty seats.

But in a different page the same image might have a different context.

Alt-text on a page about recent track try-outs:

‘Harvard Stadium with a man running up stone steps.’

Here the same photo is being used on a page about recent track try-outs. Now the focus is the man running up the stairs as an example of the try-out activities.

Alt-text on page about renovation projects:

‘Harvard Stadium with cracked concrete pillars.’

On a page about renovation projects, the same image has a completely different context and so different alt text. Now the pillars are the focus of the photo and why it's been included on the page.

Writing good alt text

"Good" alt text can be very subjective. The most you can do is to think carefully about the image and its context. And then write alt text accordingly.

However, we have some good questions that might help you.

Alt text tips
  • Consider purpose and context
  • Keep descriptions concise
  • Prioritise information
  • Use plain language

We’re going to whiz over some alt text tips now, but I’ve got a resource with more detail for you to take away.

We’ve already started thinking about the first tip. Think about why the image is there and what it’s trying to say to people? How does the image relate to its surrounding context?

Keep descriptions concise – don’t add information that can’t be seen or aren’t relevant.

Also double check any auto-generated alt text. Platforms like Facebook or even Microsoft products try to be helpful by making their best guess at what the alt text should be. Remember "4 people standing in front of a wall" could be 4 gold medallists showing off their medals, protesters with signs, or people standing in front of a firing squad.

Put the important information first. Don’t hide it at the end.

You don’t need to add words like 'image', 'icon', 'picture', because that information is already announced to screen readers. Maybe include ‘caricature’ or ‘painting’ if it’s important to know that in the context of the image.

Giving a hamburger icon the alt text "Hamburger" only makes sense to someone who knows that in this case, the hamburger is a technical term for an expanding menu. To people outside the technology sector, hamburger is something you eat. Not something you click on. Remember to apply plain language principles in alt text too.

If it’s likely that someone will be referring to your icon in instructions, make sure the alt text matches the expected name. For example an instruction like 'click on the 3 dots on the top right-hand corner' is great for visual user. But if the 3 dots on the top right-hand corner have the alt text 'Settings', a screen reader user will never find them.

Alt text and captions

Alt text: provide assistive technology users missing visual information

Captions: adds context to images for all users

Captions are there for everyone to perceive (sighted users see it, screen reader users hear it)

Don’t duplicate text for both alt text and captions.

Don‘t duplicate the caption in the alt text because this just means screen reader users hear the same information twice.

The caption isn’t there without alt text (otherwise the screen reader user wouldn’t hear that there is an image)

Alt text: ‘Empty Virgin Australia airline counter’

Caption: ‘Ticket revenue had dried up amid coronavirus travel bans’

Describing complex images

An image with more information than can be described in alt text, such as:

  • Graphs and charts
  • Maps
  • Diagrams and illustrations

Alt text: ‘Web Accessibility for Designers poster with text description below.’

Full text description: presents the same information in the image.

Complex images need all text and a full text description that describes all the meaningful information in the complex image. here is how those two are split up.

Alt text (short text alternative)

  • provides a brief description of the image
  • may reference the location of the full text description

Full text description (long text alternative)

  • Presents the same information as the image
  • Can use styles/markup (e.g. headings, links)

Alternative versions do not have to be on the same page.

You don’t have to find a way to put all that information in text on the same page. It's fine to link off to a secondary location.

Part 3: Headings

Why are heading important?

Headings:
  • Help people navigate.
  • Tell a story.
  • Tell you where you are in a progression.
  • Create groups of content.
  • Convey the structure or hierarchy of content.

Like links, getting your headings right is one of the most important things you can do to make your content accessible.

People use them for:

  • scanning or navigating around web pages quickly.
  • judging content: do I trust this? Do I want this? Am I in the right place?

Headings tell you what you’re going to get or should do.

They should set your expectations.

And they should tell a story.

If your product is a digital service, headings should tell you how far you’ve got or where you are.

Making usable headings

Headings are not just formatted text.

Don’t ‘create’ a heading by making the text bold and increasing the font size. While this will make your text looks like a heading to visual readers, you are creating an accessibility barrier for assistive technology users.

HTML headings help assistive technology users, for example screen reader users, to navigate in a meaningful way. Just as when we saw our screen reader demonstration pull up a list of links in the previous section, we can also pull up a list of headings on the page.

A level 1 heading (H1) usually indicates the beginning of the main section of content on page. It’s generally considered a good idea to only have one heading 1 on a page. However the heading 1 does not need to be the first heading on a page.

<h1>Heading level 1</h1>
<h2>Heading level 2</h2>
<h3>Heading level 3</h3>
<h4>Heading level 4</h4>
<h5>Heading level 5</h5>
<h6>Heading level 6</h6>

In this video we will see a screen reader user on the Barclays Bank website try to find out about balance transfer cards. In the top right-hand corner you can see which keystrokes the screen reader user is using to progress through the website. Note how the screen reader announces each item by what type it is. For example headings, list items, button, link, et cetera. List of one item on first page.

Then navigating the second page by jumping from heading to heading until finding the balance transfer card section. Then you'll see another way for screen ready users to navigate by bringing up the elements list.

Headings should tell a story.

Many people 'read' your page by the headings alone. This includes screen reader users, people with dyslexia, and people in a hurry scanning a page.

Test your headings by removing all the paragraph content on the page and just leave the headings in place. Would someone still get an idea of what the page was about from just the headings?

Creating structure

Headings show structure.

It is crucial that headings reflect the structure of content. The structure of headings and subheadings let people, especially screen reader users, know the intended structure of the page. Once someone knows the structure, they can navigate content by headings.

Headings usually don’t have punctuation.

Sometimes text is marked up as a heading so that it appears in larger font.

This practice is not accessible:
  • The heading doesn’t describe the information beneath it.
  • The markup is usually done for style purposes and not to convey content structure.

If you need a heading group (heading and immediate subheading) your developer can code this for you in HTML.

Put headings at the start of each new section.

A section is either a new visual region or a new idea/topic.

Having headings at the beginning of your section is a great way to break up content on a page. It is also one of the most common ways screen reader users navigate around a webpage.

Keep heading close to related text.

Remember the proximity discussion from the last session.

Choose headings for hierarchy, not style.

Writing great headings

Headings need to describe a section of content.

Headings need to be descriptive so that people can predict what they’ll find in each section of content.

Again, you can test it out by removing the body text and just leaving the headings on the page. See if other people still grasp what the page is about.

Main heading relates to page title and any links to this page.

Put the important information first.

As we’ve talked about previously, frontloading (or putting your most important information first) helps when people are scanning your page for the information they're seeking.

Keep headings short, but not mysterious.

As we’ve talked about previously with link text, alt text and plain language, you want to keep headings short and to the point. Long headings will lose people's interest and attention.

Part 4: Document titles

Why are document titles important?

<title>This is a document title</title>

A document title is not a heading.

Document titles matter.

Page titles describe the page content of each page and differentiates that page from all the others. You see them when you scan your open browser tabs, when you bookmark a page, or when your page turns up in search engine results.

Screen reader users have a shortcut that will read the page title, so they know where they are.

Document titles are descriptive.

Page titles need to be descriptive. A page title with just the company name doesn’t really tell anybody what the page is about. If you’re thinking about descriptive page titles, think about someone searching through a list of bookmarked pages to find yours.

Document titles are unique.

Most of the time page titles are unique because the page is a stand-alone page. However with multi-page processes, like a form, or with single page applications (where the page never actually refreshes) it is very common to see the same page title applied repeatedly. This is not good. A good rule of thumb is that if the main heading of the page has changed, so should the page title. This may be an issue you need to talk to your developer about.

Put important information first.

Best practice puts the title of the page before the title of the organisation.

Google truncates huge page titles at about 60 characters. If you have a long company name that appears first in the page title, the actual name of the page may never appear in Google results.

This screenshot illustrates many of the points I've been making about page titles.

While the W3C has not put their own name first, the way they've formatted the page titles and the name of the page, makes it hard to differentiate between the success criteria.

Also note that I don’t have to click on a tab to get a preview of the full page title.

Document title format is consistent.

Keep the format and style of your page titles consistent.

Best document title separators.

There isn’t really one best solution for the punctuation separator between the name of the page and the name of the organisation in your page title. However, I will point out that using the | (sometimes called a pipe) could be easily miss read as the letter L or I. So far research I’ve conducted suggests that using a hyphen or a dash is best.

<title>Page name: Site name</title>
<title>Page name | Site name</title>
<title>Page name - Site name</title>

Match your document title with the main heading of the page.

The usual best practice is to match your page title with the main heading of the page (usually the H1). Unless it conflicts with the frontloading rule. Don’t forget to put the most useful information first. Probably not the name of the organisation.

Part 5: Lists

Why are lists important?

Lists help content scanning.

Lists create groups that stand out from the rest of the content and therefore make that content more easily scanned.

  • Use numbered lists when the order of items matters
  • Use bulleted lists where there is no clear order

Lists in sentences add cognitive load. For example:

“In the winter I like to eat a variety of fruit like apples, grapefruit, plums, persimmons, mandarins, papaya, cherries, nectarines, strawberries, apricots, and raspberries to remind me that summer is coming soon.

Many tables should be lists.

This is something I'll just mention quickly, because We’ll talk more about tables later.

Tables are often used when a list is the better option.

If the data isn't relational data, and usually if the data isn't quantitative data, it might make a better list.

Part 6: Tables

Table basics

Tables convey relational information in a way that’s easier to read.

Tables rely on the relationship between data in the table and they reference names in the column or row headings. In order for these to be accessible those heading cells need to be coded as such.

Every column (and sometimes row) needs a header cell.

This must be coded (see next page). Header cells are necessaru for association.

<th>This is a header cell</th>

Never leave header cells blank.

In a moment you'll see a screen reader reading through a table. If the table header cell is left blank, it causes big problems.

Screen reader jumps straight from the heading Example 1, etc to table and then interrogates the table structure.

Also note the name of the table “Concerts” isn’t read out because it hasn’t been associated with the table as a caption.

Table information is read out first. Table with 4 rows and 3 columns. Then row number and column number.

Once we get to the second row, past the headings, note that the heading value is read out before the cell value.

This is why you shouldn't leave table header cells blank.

Avoid adding headings inside tables.

Headings, like the type we just talked about in the heading section, are great for navigation. However, inside tables they are confusing because of the way screen reader users navigate pages scanning for headings. When the heading that they’ve jumped to happens to be inserted in a table, it gets a bit confusing what is going on.

Table captions

<caption>Great cities to visit</caption>

A caption is better than a heading because it programmatically associates the table with the name and will also generate an accessible name.

Example: if the user hits T and jumps to a table it will read the caption but it would skip past the heading.

Similarly if a user pulls up a tree of all the page elements it will have the table name which wouldn’t happen with a heading.

Associate table captions with the related table.

Adding a caption to a table tells people visually what the table is about. If the caption is coded to associate it with the table, then screen reader users get that relationship too.

Great cities to visit
France Italy Spain
Paris Florence Barcelona
Nice Rome Madrid
Carcassone Venice Seville

Simple vs. complex data tables

Simple tables have column headings and may also have row headings.

Complex tables, or irregular tables, may have heading cells that span multiple columns or rows. Or they may have data cells associated with multiple headings.

Simple tables are also easier to understand because they usually convey one topic at a time.

Make complex tables simple

While you can code complex tables that have multiple column and row headings, it's easier to convert them into simpler tables. Break up complex tables into simpler tables if you can.

Other table considerations

Use tables for data rather than layout, except for email layout.

Layout tables were common in the 1990s before the invention of CSS, to prevent strange layouts.

It is possible to make them accessible, but your developer will need to do some coding to make it work for everyone.

[Exception is email layout because CSS doesn’t always apply]

Avoid nested tables.

A nested table is a table within a table. These are at best a bad idea because it’s hard to work out where you are and what things relate to what headers. At worst, one of my colleagues who uses a screen reader, compared it to reading in 3D.

Tables can be accessible to screen readers.

Tables are very accessible, if they’re coded correctly. As you saw in the image is module, accessible tables can provide an alternative to inaccessible data.
